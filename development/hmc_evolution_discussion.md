# HMC 進化方向性の議論メモ

> 作成日：2026-05-02
> 議論の相手：Claude Sonnet 4.6

---

## 前提：現状のHMC評価

### 強み
- dry-runカルチャーが徹底している（壊さない設計）
- SKILL境界が明確で、追加・修正がしやすい
- 経理・HR・SNS・タスク管理と業務の主要領域を網羅
- Geminiによるマルチモーダル入力（OCR）で紙の書類も取り込める

### 構造的な弱点
**全SKILLがリアクティブ**——呼ばれるまで何も起きない。
現代のAI経営支援ツールは「Proactive Agent」方向に進化している。

---

## 改善の3方向

### 方向1：プロアクティブな監視・通知

毎朝ブリーフィング時に全SKILLのシグナルを自動チェックする層を追加する。

```
毎朝ブリーフィング時に自動チェック：
- 未処理の請求書メール（invoice_processor）
- 7日以内に期限が来るタスク
- CF残高が閾値を下回ったら警告
- 処理されていない経費データがあれば通知
```

### 方向2：SKILLをまたいだ文脈の統合

各SKILLは独立して動作しているが、横断的な視点がない。
例：「5月の企画プッシュ → SNS告知 → 売上への影響」を一貫して追えない。

**具体的提案：** `finance_analyzer`に「プログラム別分析モード」を追加。
売上 × SNSエンゲージメント × シフトコストを組み合わせて、どの企画が利益を出しているかを可視化する。

### 方向3：モバイル・非同期入力チャネル（LINEボット）

現状は全操作がデスクトップターミナル前提。
原っぱ大学はフィールド事業なので、野外での即時入力が必要。

---

## 方向3 詳細：LINEボット設計

### アーキテクチャ

```
LINEグループ
    ↓ HTTPS webhook
XserverVPS（Dockerコンテナ：Python FastAPI）
  ├── LINE署名検証
  ├── メッセージ受信・保存（SQLite or JSONファイル）
  └── GET /queue エンドポイント（HMCが取りに来る）
    ↓ HMC起動時にpolling
ローカルHMC（WSL）
    → 分類・各SKILLへ振り分け
```

### なぜGAS（Google Apps Script）ではなくVPS？
- GASはコードのコピペ修正が手間
- Xserver VPSはすでに稼働中でDockerが動いている
- HMCと同じPythonで全部書ける

### VPS側で作るもの（Python FastAPI、50〜80行程度）

```
POST /webhook  # LINEからメッセージ受信 → DB保存
GET /queue     # HMCが未処理メッセージ一覧を取得
POST /ack      # HMCが処理完了を通知 → 既読フラグ
```

### HMC側で作るもの
- `hmc_pilot`に朝ブリーフィング時のpollingロジックを追加
- VPSのキューを取得 → タスク/経費/SNSネタに分類

### 写真・画像の扱い
- LINE画像 → FastAPIがDriveに保存
- expense_processorのinputフォルダに誘導

### 事前確認事項
1. VPSにdocker-compose.ymlがあるか
2. VPSにドメイン/SSL証明書が設定済みか（LINEはHTTPS必須）
3. LINE Messaging APIアカウントが作成済みか

## プロアクティブAIの最新パターン

### 3つの層

| Layer | 説明 | 例 |
|---|---|---|
| Layer 1 | Time-triggered（cron） | 毎朝7時にCF確認 |
| Layer 2 | Event-triggered（条件駆動） | 請求書メールが届いたら自動処理 |
| Layer 3 | Ambient Agent | 複数シグナルを合成して「今これが重要」と判断 |

### 最新事例（2024〜2025）
- **Linear**：新issueをAIが自動トリアージ、スプリント終盤に「終わらないかも」と先出し
- **Notion AI**：「このタスク3週間更新なし」と自動提案
- **Salesforce Einstein**：商談のメール返信が3日ないと「リードが冷えています」と通知

### HMCへの適用案

| トリガー | アクション |
|---|---|
| 毎週月曜 | SNS週次コンテンツ計画の準備を促す |
| 月末3日前 | 未処理の経費・請求書があれば警告 |
| CF残高が閾値以下 | 翌週の支払い予定と照合してアラート |
| SNS投稿2日後 | エンゲージメントレポートを自動生成 |
| バックログに期限切れタスクが3件以上 | 朝ブリーフィングで先頭に出す |

### HMCへの実装方針
HMCはローカル起動のため、Layer 3（常時監視）より「起動したときに賢く動く」設計が現実的。
VPSが使えるなら、そこでcronを動かしてLINEに通知するLayer 2も自然に作れる。

## チームとの協働へ

### 現状と将来の比較

**今（個人ツール）**
```
塚越さん → HMC → 実行・報告
```

**LINE + Layer 2 後（チーム協働）**
```
チームメンバー → LINEグループ → HMC → 塚越さんに集約・報告
                                    ↓
                             HMCがチームにも能動的に通知
```

### 原っぱ大学チームでの具体例

**チーム → HMC（入力）**
- 和田さんが現場写真をLINEに投稿 → sns_pilotのネタ候補に自動追加
- スタッフがシフト希望をLINEに書く → shift_managerが取り込む
- 「藤澤さん支払い完了」とLINEに書く → hmc_pilotがタスク完了処理

**HMC → チーム（能動通知）**
- イベント前日に「明日の企画準備チェックリスト」をLINEに自動送信
- 月初に「今月のシフトアンケートを送ってください」と担当者に通知
- 経費精算締切3日前に「未提出の人はLINEで写真送って」と促す

### 重要な設計判断：情報の境界線

```
【塚越さん専用レイヤー】経営・財務・意思決定  ← 今のHMC
          ↕ 情報の境界線
【チーム共有レイヤー】シフト・SNS・イベント準備
```

財務情報はチームに見せない、通知の内容と宛先を明示的に設計する。

## 実装ロードマップ（優先度順）

| ステップ | 内容 | 難易度 | インパクト |
|---|---|---|---|
| 1 | VPS + LINEボット（個人→チーム入力） | 中 | 高 |
| 2 | Layer 2 proactive通知（HMC→チームへの能動発信） | 中 | 高 |
| 3 | 朝ブリーフィングへの全SKILLシグナル統合 | 低 | 高 |
| 4 | プログラム別収益分析（finance_analyzer拡張） | 中 | 高 |
| 5 | 情報境界の設計（塚越専用 vs チーム共有） | 低 | 中 |
