# (B) 当月の流れ(シフト中心)

> 毎月の **月末・月初・月中** に走るシフト周りのルーチン。`shift_manager` SKILL のタスクと一対一に近い対応関係。

**注**: 各ステップは「**目的 (Purpose)**」と「**現状の方法 (Current Method)**」を分けて記述している。目的は不変、方法は改善対象。詳細は [README.md](README.md) と [ADR](../../../docs/decisions/2026-05-24-workflows-as-truth-and-improvement-targets.md)。

## 全体図

```
月末(最終フィールド後) ─ 当月稼働表の作成(放サボ稼働の手入力含む)
            │
当月1日  ─ ① 翌々月シフトアンケート送信(staff LINE)
            ② 前月稼働の確認依頼送信(staff LINE) → 締切10日
                │
                ├─ 業務委託 → 請求書を作成・メール送付(10日まで)
                ├─ アルバイト → freee で経費精算(10日まで)
                └─ 外部スタッフ → 交通費を LINE コメントで申請(10日まで)
            │
当月10日 ─ ① 翌々月シフト確定 / ② 稼働確認締切 = 精算データ集約
            │
適宜    ─ 翌月の担当プログラムの LINE グループを作成(都度)
```

## ステップ

### 1. 月末 — 当月稼働表の作成

- **時期**: 当月最後のフィールド開催後 〜 月末
- **担当**: シフト管理担当(現状: TODO 確認)
- **目的**: 翌月1日の「稼働確認依頼」で使う **正本の稼働表** を完成させる
  - 必須データ: 全フィールド開催の正確な稼働時間 + 放サボ稼働時間
- **現状の方法**:
  1. [月次カレンダー Sheet](https://docs.google.com/spreadsheets/d/1_RMAQuSb3eWV30WGQ_gsJI5M6Ll1WHvbM4ifkGDuNkM/) を点検
     - 開催されなかった日程が残っていないか
     - 稼働時間とスタッフ名の誤りがないか
     - 突き合わせ先: [Notion フィールドレポート](https://www.notion.so/5dab98a40ae443849e3804c0b431abe2) + 運営LINE
  2. `shift_manager` SKILL の `generate_working_hours.py --month YYYY-MM` で [稼働表 Sheet](https://docs.google.com/spreadsheets/d/1nevys4etwvn4NQToetsj6GLAVrtJT4Y4rj5pn30yZ84/) に展開
  3. [[kodomon]] から放サボ勤怠 CSV をエクスポートし、稼働表に **手入力** で反映
- **アウトプット**: 当月稼働表(`YYYY-MM_稼働時間` タブ)が完成

### 2. 月初1日 — 翌々月シフトアンケート送信 + 前月稼働の確認依頼

- **時期**: 毎月1日(朝)
- **担当**: シフト管理担当(現状: TODO 確認)
- **目的**: 2系統が同時に発火
  - **(a)** 翌々月の人員を集めるためのアンケートをスタッフに開放する
  - **(b)** 前月稼働をスタッフ各人に確認させ、精算ルートを走らせる(締切10日)
- **現状の方法**:
  - **(a) 翌々月シフトアンケート送信**
    1. `shift_manager` SKILL の `generate_shift_form.py --month YYYY-MM`(翌々月分)で Google フォーム生成
    2. フォーム URL をスタッフ LINE グループに人手で投稿
  - **(b) 前月稼働の確認依頼送信**
    1. ステップ1 で完成した稼働表のスクショを撮る
    2. スタッフ LINE グループに人手で投稿 +「自分の稼働を確認して」と一斉案内
    3. スタッフが自身の `PaymentType` に応じて 3 ルートで動く:

      | PaymentType | スタッフの動き | 締切 |
      |---|---|---|
      | `業務委託` | 請求書を作成 → 原っぱ大学にメール送付 | 10日 |
      | `給与`(アルバイト) | freee で経費精算 | 10日 |
      | `追加`(外部・請求書なし) | 交通費を LINE コメントで申請 | 10日 |

- **アウトプット**:
  - 翌々月シフト募集のフォームが公開状態
  - スタッフ全員が前月稼働を確認できる状態 + 精算アクションへの動線

#### 改善余地(Improvement Hints)

| # | 現状の方法 | 改善余地 | ステータス |
|---|---|---|---|
| 💡 | (a) フォーム URL の LINE 投稿が人手 | ガクコ `/send (group=staff, approval=true)` で剪定経由配信(セッション4 ADR 決定6 の実装) | **着手予定**(種1) |
| ❓ | (a) フォーム生成の前提に「Q列チェック」が要る | 月次シートの Q列チェックを誰がいつ入れるかが未明示。種化の前に確定要 | 未検証 |
| ❓ | (b) スタッフへの見せ方「スクショ + 一斉案内」 | セッション4 ADR で保留中。候補: 個別テキスト要約 / スプシ個人タブ / 現状スクショ | **検討中**(塚越さん) |
| 💡 | (b) 全体 | 種1(アンケート)と種3(稼働確認)を別の種で実装(同タイミングだが責務分離) | 着手予定(種1先行) |

### 3. 月初10日 — 翌々月シフト確定 + 稼働確認締切

- **時期**: 毎月10日
- **担当**: シフト管理担当
- **目的**:
  - 翌々月の **シフト人員配置を確定** させる
  - 前月稼働の **精算データを揃える**(請求書・経費精算・交通費申請の集約)
- **現状の方法**:
  1. `shift_manager` SKILL の `aggregate_responses.py --month YYYY-MM`(翌々月分)で回答を集計し、シフト調整シートを生成
  2. シフト調整シート上で人手で確定作業
  3. 稼働確認の 3 ルート(業務委託請求書 / アルバイト freee 精算 / 外部スタッフ LINE 申請)の到着を確認・集約
- **アウトプット**:
  - 翌々月の各プログラム現場責任者・フィールドスタッフ・フォトグラファー = 確定
  - 前月の精算データ = 揃う(以降の Freee 登録フローへ)

#### 改善余地(Improvement Hints)

| # | 現状の方法 | 改善余地 | ステータス |
|---|---|---|---|
| ❓ | 2 の確定作業(シフト調整シート上の人手) | 確定作業の自動化余地は未調査(複雑な人員配置最適化はまず人手が現実的か) | 未検証 |
| 💡 | 3 の精算データ集約 | 業務委託請求書は `invoice_processor` SKILL で処理可。3ルートの状態追跡を可視化する種があると良い | 構想中 |
| ❓ | 全体 | 「10日締切」が守られなかった場合のリカバリ手順が未定義 | 未検証 |

### 4. 適宜 — 翌月担当プログラムの LINE グループ作成

- **時期**: 都度(翌月の運用が始まる前まで)
- **担当**: 各プログラムの現場責任者
- **目的**: プログラム毎にスタッフ・チームが異なるため、開催毎のコミュニケーション基盤を用意する
- **現状の方法**: 現場責任者が手動で LINE グループを作成 → 関係スタッフを招待
- **アウトプット**: 各プログラムのスタッフコミュニケーション基盤
- **次の遷移**: 個別開催の運用は [[program-execution]] へ

#### 改善余地(Improvement Hints)

| # | 現状の方法 | 改善余地 | ステータス |
|---|---|---|---|
| ❓ | LINE グループ作成は手動 | LINE Bot API でグループ作成・招待を半自動化できる可能性(LINE側仕様調査要) | 未検証 |
| 💡 | 確定シフトからプログラム単位の参加者を自動抽出 | `shift_manager` の出力からプログラム × スタッフのマトリクスを生成 → グループ作成時にコピペ用テキストを下書きするだけでも価値あり | 着手可能 |

## 不変条件(invariants)

- **シフトは「翌々月」を確定させる**(N月の1日には N+2月分のアンケート、10日には N+2月分の確定)
- **稼働確認は前月分を月初〜10日で集約**(請求書・経費精算・LINE申請の3ルート)
- LINE グループは **プログラム毎** に作る(プログラム毎にチームが組み変わる)
- 稼働表は **当月最後のフィールド後 〜 翌月1日朝までに完成** している(翌月1日の確認依頼の前提)
