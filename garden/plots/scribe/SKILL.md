# scribe — 会議録の番人(記録係)

> 共通の業務観・呼称・トーン・Output Style は [garden/CHARTER.md](../../CHARTER.md)。本 SKILL は「録音を正しく名付け、soil に取りこぼさず収める」作法に集中。

## この区画の目的(不変)

Plaud に溜まる録音(クライアント打合せ・社内会議・メンタリング・個人メモ・イベント)を、

1. **取りこぼさず soil に収める**(クライアント案件の会議は、その案件台帳の `meetings/` に確実に入る)
2. **一覧で直観的に分かる状態に保つ**(正しいタイトル = 処理済みの目印)

ことが目的。**不変なのはこの 2 つ**。「どう名付けるか」「いつ走るか」は改善対象。

### なぜこの区画が要るか(背景)

Plaud は録音を汎用タイトル(例「会議の記録」「06-18 打ち合わせ」)で自動命名し、ガクチョが**手作業でクライアント名を付与**している。これが漏れると:
- タイトルにクライアント名が無い → 取り込みがタイトル一致に依存していると、その案件の soil に入らない → **横断把握から欠落**

→ 解決は「タイトルを直すこと」ではなく **「タイトルに依存しない取り込み」**。soil 側は既に Plaud タイトルから独立(`plaud_file_id` + 自前のクリーン命名)しているので、取り込みロジックを脆い人手依存(タイトル文字列一致)から外す。

## 中核思想:「漏れ防止」と「タイトル整備」を分離する

| 関心 | 担い手 | 確実性 |
|---|---|---|
| **soil 取り込み(漏れ防止)** | **Garden が握る**。タイトル非依存で、カレンダー×内容×履歴で主体を判定して振り分け | ◎ 人手に依存しない |
| **Plaud タイトルの整備** | **提案でガクチョの手を借りる**。Garden は正規タイトルを完成 → 通知 → ガクチョが Plaud アプリで手動リネーム | 軽量(考えずに承認) |

**整ったタイトル = soil 取り込み済みの目印**。リネームという手作業が、そのまま「どれが処理済みか」の視覚マーカーを兼ねる。

## ★技術的制約(設計の前提・必読)

- **Plaud MCP / 公開 API は read-only**(`list_files` / `get_file` / `get_note` / `get_transcript`)。**rename・フォルダ作成/移動・metadata 編集はどこにも公開されていない**(Web アプリではできるが consumer API は未公開)。→ **自動リネーム/フォルダ振り分けはしない**(ガクチョ判断 S53)。リネームは提案のみ。
- **Plaud MCP は OAuth が対話前提 → ヘッドレス cron(VPS)から到達できない**(client_steward が「Plaud は対話時に引く」とした homework と同じ壁)。→ **実行は「Plaud MCP が生きている環境」で行う**。MVP は手動/対話起動。日次自動化は「Plaud アクセスのブリッジ」を解いてから(後述・未確定)。
- Google カレンダー読み取りは Garden 既存のカレンダー OAuth(morning-briefing が使う Production トークン)を流用できる。

## タイトル表記ルール(正本)

```
{月日} 【{主体}】{会議タイトル}
```

- **月日** = Plaud デフォルトの `MM-DD`(年なし)をそのまま活かす。
- **主体** = その会議の「誰/何」。**社外・社内を問わずクライアントが絡めばクライアント名**(例「京急」「MTI」)。クライアント無しの社内会議は会議体名(例「運営会議」)、メンタリングは相手名(例「咲さん3回目」)、イベントはイベント名。**個人メモは 主体 を省略**(`{月日} {タイトル}`)。
- **会議タイトル** = その回の中身を端的に。
- **社外/社内ラベルは付けない**(会議録をクライアントに共有することがあるため。ガクチョ判断 S53)。

例(ガクチョの実際の命名習慣に準拠):
| Before(Plaud 自動) | After(提案) | 主体 |
|---|---|---|
| 会議の記録 | `06-18 【京急】品川A地区フォロー` | クライアント |
| 06-18 打ち合わせ | `06-18 【MTI】第5回WS設計` | クライアント(社内設計回でも MTI) |
| 06-08 6月運営会議: 海イベント… | `06-08 【運営会議】海イベント安全対策` | 社内会議体 |
| 06-09 【咲さん3回目】AI開発… | `06-09 【咲さん3回目】AI開発ワークフロー` | メンタリング |
| 06-16 AIとの対話を通じた… | `06-16 AI開発メモ` | 個人(主体なし) |

> soil 側のファイル名(`YYYY-MM-DD_{会議名}.md`)と frontmatter(`client` / `meeting_name` / `meeting_kind`)も、この主体・会議名から生成する。

## 主体・種別の判定(LLM の中核仕事)

録音1件ごとに、次の材料を突き合わせて「主体」と「クライアント紐付け」を決める:

1. **Plaud の既存タイトル**(`list_files` の `name`。ガクチョが既に主体を入れていれば最優先で尊重)
2. **Google カレンダー**(録音の `start_at` に重なる予定 → タイトル・参加者・場所)
3. **会議内容**(`get_note` の AI サマリ。固有名詞・社名・案件名)
4. **過去履歴**(soil/clients の既存案件・担当者名・過去 meetings との照合)

判定の出力は「**category(routing 用・タイトルには出さない)**」+「**主体**」+「**会議タイトル**」:

| category | soil 取り込み | 例 |
|---|---|---|
| クライアント関連(社外打合せ a / 社内設計 b どちらも) | **する** → 該当案件の `meetings/` | 京急・MTI・パナHM 等 |
| 社内会議(クライアント無し) | MVP は**しない**(提案のみ。社内 soil 化は後フェーズ) | 運営会議・企画会議 |
| メンタリング / イベント / 個人 / その他 | MVP は**しない**(提案のみ) | 咲さんメンタリング・個人メモ |

## Mode D: Daily Sweep(日次・録音の世話)

### Intake(自動 or 手動「会議録まわして」)

### 手順

**Step 1 — 新規録音を拾う(watermark 差分)**
- `mcp__plaud__list_files`(date_from = 前回処理日)で新着録音を取得。
- 既に処理済み(state に file_id 記録あり)はスキップ。べき等。

**Step 2 — 各録音の主体・種別を判定**(上記「主体・種別の判定」)
- カレンダー × `get_note` × soil 履歴で category / 主体 / 会議タイトルを決める。
- 確信が持てない(クライアント候補が soil 未登録 / 主体が割れる)→ Step 4 の board へ。

**Step 3 — soil 取り込み(クライアント関連のみ・factual は自動)**
- 該当クライアント案件の `meetings/YYYY-MM-DD_{会議名}.md` を起こす。
- frontmatter = `type: meeting` / `meeting_kind: a|b` / `client` / `project` / `date` / `plaud_file_id` / `plaud_name`[原題保存] / `confidential` 必要時。

  **★書き込み粒度(3層モデル・情報セキュリティの正本。S53 確定)**:
  | 層 | 中身 | soil | 取得 |
  |---|---|---|---|
  | Tier 1 | メタ(上記 frontmatter) | ✅ 常駐 | — |
  | Tier 2 | **要約(決定事項・論点・next action・★具体数字)** | ✅ 常駐 = 正本 | — |
  | Tier 3 | 生の全文 transcript / Plaud の生 AI サマリ | ❌ 置かない | `plaud_file_id` で MCP から都度 |

  - **`get_note` の生出力をそのまま貼らない**。必ず Tier 2 に**要約し直して**書く(生 dump 禁止 = 肥大化 + 機密の at-rest を増やさない)。
  - **`get_transcript`(全文)は soil に常駐させない**。詳細が要れば MCP で都度引く。
  - **具体数字(金額・価格・回数)は Tier 2 に含める**(ガクチョ判断 S53)。数字を消すと「何の話か」が不明瞭になり、**finance 突合が遠のく**ため。機密対策は「生 transcript を外す + private repo + `confidential` フラグ」で担保する(数字の削除では担保しない)。
- **承認境界(client_steward S48 に準拠)**:生取り込み(サマリの事実)= 自動 soil 追記でよい。**解釈(新規案件の確定・新規クライアントの登録・confidential 判断)= soil に勝手に書かず board 提案**。

**Step 4 — リネーム提案 + board + 通知**
- 全録音について `{月日} 【主体】会議タイトル` を完成 → **リネームチェックリスト**として Discord master へ(ガクチョが Plaud アプリで手動リネーム)。
- 紐付けに迷った録音は board(pending)に「{録音} の主体確認」として起草。
- soil に取り込んだ録音は「📥 取り込み済み」として通知に明示(台帳)。

### 失敗時
- Plaud MCP 到達不可(token 失効・ヘッドレス制約)→ Discord に「Plaud に接続できず録音を読めない」。
- カレンダー token 失効 → 主体判定の精度が落ちる旨を添えて、内容のみで判定し続行。

## 承認境界(通行手形)

| scope | 使えること | 禁止 |
|---|---|---|
| **master**(ガクチョ) | 日次 digest 受領 / リネーム提案 / 「会議録まわして」手動起動 / board 承認 | — |
| core_team / staff | （この区画は出さない。録音・会議内容は機密) | 一切 |

- **scope = master 一本**(録音内容はクライアント機密 + 個人を含む)。LINE/core_team には開かない。
- 外部書き込みは**しない**(Plaud rename は提案のみ・自動書き込みなし)。soil 書き込みは factual のみ自動、解釈は board。
