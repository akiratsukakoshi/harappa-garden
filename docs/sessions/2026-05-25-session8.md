## 流れ

2. 4 本の draft を起草:
   - `daily-pilot/recurring-spawn.md`(cron 06:25、Garden 内完結)

   - `daily-pilot/night-review.md`(cron 22:30、active クリア + 反映)
   - `daily-pilot/inbox-process.md`(event、inbox 振り分け)
3. draft 過程でスキーマ拡張 5 項目を新設(seeds/README.md 反映済):
   - `pruning.channel: none`(自律完結種)
   - `on_complete`(剪定なしで完了報告のみ送る種用)
   - `trigger.exclude`(event 種の watch 除外 glob)
   - `trigger.debounce`(event 種の連続書き込み待ち)

4. 塚越さんから本質的指摘: **「monthly タスクは完了して archive に入った後、同期間内で再 spawn されないか?」**
5. 3 つの問題シナリオを整理:
   - 同日 2 回起動時の再 spawn 可能性(backlog から消えた状態を未追加と誤判定)
   - 「月初 N 営業日」等の複数日許容 period での再 spawn 可能性
   - タスク名の表記揺れによるマッチ失敗
6. 対応 3 案を提示(案 B: 別ファイル / 案 C: master 側 last_completed / 案 E: backlog+archive 両方 grep)→ **塚越さん案 E 採択**
7. 案 E で改訂:
   - recurring-spawn の prompt をマーカー方式に書き換え(`<!-- recur:{id}@{period_id} -->`)
   - recurring_master のスキーマ要件(ID 必須・period 表現)を本文に追記
   - night-review に「archive 転記時の元行完全保持」を明示

### 種スキーマ拡張 5 項目を導入(暫定)

| フィールド | 用途 | 初出種 |
|---|---|---|
| `pruning.channel: none` | 自律完結種(剪定なし)。approver/notify は null | recurring-spawn / night-review / inbox-process |
| `on_complete` | 剪定なしで完了報告だけ送る種用 | night-review |
| `trigger.exclude` | event 種の watch 除外 glob | inbox-process |
| `trigger.debounce` | event 種の連続書き込みファイナル待ち | inbox-process |
| `{event.path}` 変数 | watcher daemon が渡すマッチファイルパス | inbox-process |

[seeds/README.md](../../garden/seeds/README.md) の「スキーマ拡張メモ」表に集約。

### 案 E: recurring 完了済み再 spawn 防止の仕組み

- **キー** = `recur:{id}@{period_id}`(例: `recur:r001@2026-06`)
- **マーカー** = `<!-- recur:{id}@{period_id} -->` を backlog 行末に埋め込む
- **判定** = recurring-spawn が backlog + archive 両方を grep し、既出なら skip
- **前提** = recurring_master の各エントリに `id:` 必須 + night-review が `[x]` → archive 転記時に元行完全保持

### recurring_master.md スキーマ要件(暫定)

```yaml
- id: r001 必須
- name
- period: daily | weekly | monthly | yearly
- day / weekday / month_day
- category
- description
```

詳細は [recurring-spawn.md](../../garden/seeds/daily-pilot/recurring-spawn.md) 本文「recurring_master.md スキーマ要件(暫定)」セクション参照。
