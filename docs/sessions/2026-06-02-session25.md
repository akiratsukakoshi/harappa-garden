## 2. 構造ガード実装(`send_pending.py` 連続失敗ガード = S25 機能)

「失敗ファイルが永遠に再試行される構造そのものが地雷」というガクチョの判断で、根本対策として `send_pending.py` を拡張:

### 設計

- board frontmatter に `fail_count` / `last_fail_at` / `last_fail_reason` を upsert
- 2 回目以降の失敗は `❌` 通知を **抑制**(`notify_master` が prefix を見て自動 sink、`_LAST_FAIL_REASON` 内部キャプチャは継続)
- `SEND_PENDING_FAIL_THRESHOLD`(default **3**)回失敗した `status: approved` board は `garden/board/failed/{name}.FAILED.md` に **自動退避** + `⚠️` 通知を 1 回だけ
- `status: test` は庭師の試行錯誤フェーズなので **隔離しない**、通知抑制のみ
- 成功時は fail 系 frontmatter を掃除して clean に

### 実装と検証

- 構造: 既存の `dispatch_*` 関数は無改修。`notify_master` を prefix 検出するように 1 関数だけ拡張 + ヘルパ 4 つ追加(`get_fail_count` / `write_fail_state` / `clear_fail_state` / `quarantine_to_failed`)+ `process_one` を fail-count 連動に書き換え
- /tmp に独立 fixture を作って 4 ラウンド実行 → **Run1: ❌ 1 通 / Run2: 抑制 / Run3: ⚠️ 隔離通知 / Run4: 何もせず** を確認
- VPS に scp デプロイ + venv で py_compile 構文 OK

→ 通知総数: 最大 **2 通(初回 + 隔離)** で打ち止め。spam 構造の根絶。

→ 明朝 06:30 cron で挙動変化を確認。

### 副題: ガクコは自分のロジックを変えられるか?

ガクチョの問い「ガクコは自らフィルタを変えられる?」への答えは **No**。

ガクコは cron 起動の `claude -p` 単発プロセスで、その時点で読む SKILL.md / seed.md に従ってファイル操作するだけ。SKILL 自体を能動的に書き換える設計にはなっていない(意図して与えていない)。ロジック修正は Garden レポジトリの直接編集 = ガクチョ + Claude Code セッションで行う、という今の構造が確認された。

## 7. 設計判断のメモ

### Workflow ツール起動の自制

セッション中盤、ガクチョのメッセージで「運用フロー」というキーワードが含まれていたため `Workflow` ツール起動の system reminder が発火した。が、文脈は「手順(procedural flow)を定める」意味であり、多エージェント並列を必要としない authoring 作業と判断 → 起動せず通常の Edit/Write/Read で完了。**システム自動 reminder と人間の意図の区別** は今後も意識する。

### 測量士の独立性確保

codex への入力は MAP + 最新 session を最小単位とし、Claude Code の議論履歴を渡しすぎない方針で運用ルールに明記。外部視点の独立性が損なわれるリスクへの対処。

### 「外部視点」を Garden 語彙にどう収めるか

測量士は植物・番人・菌糸・草木の精とは別系統の役割。語彙表に混ぜると比喩が崩れるため、 **「外部視点(庭の中の役者ではない)」** という新セクションを作って分離 → Garden の生態系比喩は維持しつつ、外部観測者の位置づけを明示。
