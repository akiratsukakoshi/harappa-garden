## 3. SKILL 動的再読み込み(S27 持ち越し消化)

### 3.1 問題と解決方針

現状の bot.py 起動時挙動:
- CHARTER / SKILL / PERSONA を起動時 1 回だけ load(open().read() で定数化)
- Stage C で追加した MEMORY_STATIC も起動時のみ
- 2 分毎 keepalive は「プロセスが落ちてない限り何もしない」→ 明示的に kill しないと再起動されない
- 結果: CHARTER / SKILL / memory wiki / 過去 RAW を編集しても **bot 再起動まで反映されない**

解決: **mtime ベースの動的キャッシュ**。読み込んだ時の mtime を覚えておき、ファイル mtime が新しくなっていたら再 Read。

### 3.2 実装(3 つのキャッシュクラス)

[bot.py](../../garden/services/garden-gaku-co/bot.py):

| クラス | 用途 |
|---|---|
| `_FileCache` | 単一ファイル mtime check → persona / charter / skill |
| `_DirCache` | ディレクトリ内最大 mtime check(`index_first` で index.md を先頭固定)→ memory wiki |
| `_MemoryPastRawCache` | 対象日 + ファイル mtime check(日またぎ自動対応 + ingest-raw による frontmatter 書き換えにも追従)→ 過去 3 日 RAW |

書き換え:
- 旧定数 `CHARTER` / `SKILL` / `PERSONA` / `MEMORY_STATIC` を全廃
- `build_dialogue_prompt` で `_charter_cache.get()` / `_skill_cache.get()` / `_memory_wiki_cache.get()` / `_memory_past_raw_cache.get(now.date())` を呼ぶ形に
- `run_claude` で `_persona_cache.get()` を渡す
- 起動時ウォームアップで 5 つのキャッシュを事前ロード(初回 turn の遅延回避)

### 3.3 検証

```
1. ローカル syntax check OK
2. scp → bot 再起動 → log: bot online、warmup 失敗なし
3. Python 経由で 5 キャッシュのサイズ確認:
   persona chars:   826
   charter chars:  3839
   skill chars:    9817
   wiki chars:     3826
   past_raw chars: 4039
   today_raw chars:   0(まだ今日の対話なし)
   → 全キャッシュ正常ロード、合計 ~22KB(MEMORY_MAX_CHARS 30000 内)
```

### 3.4 効果

- **CHARTER / SKILL / PERSONA を編集** → 次 turn で反映(bot 再起動不要)
- **memory wiki 更新**(03:30 ingest-raw / 03:50 consolidate-wiki)→ 次 turn で反映
- **過去 RAW 集合が変わる**(日またぎ / archive 移動)→ 次 turn で再計算

Stage C の MEMORY_STATIC も同じ機構に乗ったため、**永続記憶 Stage B/C の結果は bot 再起動を待たず次 turn で反映**(対称性確保)。

## 4. 学び

- **永続記憶の構造が「対」+「縦」で揃った節目**:S26(soil 正本ルール)+ S29(memory 正本ルール)= 横の対称性、S22(Stage A)+ S26(A.5)+ S30(B + C)= 縦の段階。次の「Stage D = LINE 統合 / 投影ビュー / マスター透視権」までは現状の基盤で問題なく動く
- **UX 先行プロセスは「論点 5 つを箇条書き → 推奨提示 → ガクチョ OK」が安定パターン**:Stage B/C で同じパターンが 2 回成立。論点の数は 5 ぐらいがちょうど良い(多すぎると認知負荷、少なすぎると詰めが甘い)
- **A → B → C 構成と同じ「重→軽」の順は今回も快適に回った**:Stage B(重・設計議論 + 実装 + crontab)→ Stage C(中・bot.py 単一ファイル拡張)→ SKILL 動的再読み込み(軽・mtime キャッシュ機構)。S29 の A/B/C と同じ落差で認知負荷の山が緩やか
- **launcher 1.0.1 の `--model` 対応は副次的に他の種にも効く**:ingest-raw の改善余地 #1「claude-haiku-4-5 直接呼び出しでコスト 1/10」が将来の seed 編集だけで実現可能になった(SKILL Mode 1 の振り分け規約が haiku で再現できるか別途検証は必要)
- **append-only 厳格化(Karpathy との差分明文化)が Garden 固有の判断**:Karpathy LLM Wiki は LLM が本文を直接編集 / 上書きするが、Garden の memory wiki は **庭師の判断履歴** なので本文不変。整理は index と log だけ。これを SKILL Mode 5 内の注記で明示しておくことで将来の判断ブレを防ぐ
