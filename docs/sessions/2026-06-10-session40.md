## Part 2: シフト集計の 8日化 + 手動ルート(ガクチョ指示)

**確定した月内サイクル**: 1日 アンケート開始 → **7日 回答締切** → **8日 とりまとめ** → 10日 シフト確定

| 変更 | 内容 |
|---|---|
| finalize 種 | cron `0 8 8 * *`(毎月8日 08:00)に変更し **VPS crontab 登録済(初回 7/8)**。board に「遅れ回答は再集計可」を明記 |
| survey 種 v3 | 配信文の**回答期限を「配信月の7日」に固定**。`deadline_jp: "$(date +%-m/7)"` で機械計算(dry-run で `6/7` 解決確認)+ prompt に「10日以降を期限にしない」 |
| SKILL | Mode 4 を「月初8日 集計 → 10日確定」に改題、起動 2 ルート(cron + 手動)。Mode 2 に期限ルール。Mode 5 キーワードに「シフト集計」「シフト確定」 |
| workflow | [monthly-cycle.md](../../garden/soil/workflows/monthly-cycle.md) の全体図・ステップ・不変条件に 7日締切/8日集計を反映(soil 編集 → push-to-vps 済) |

**手動実施(遅れ回答対応)= bot Route A の shift 版**:
- [bot.py](../../garden/services/garden-gaku-co/bot.py) に `SHIFT_AGG_TOPICS` 話題検知ブロックを追加(expense と同型)。
  Discord master で「**シフト集計まわして**」→ ガクコが scoped Bash で `aggregate_responses.py --month {対象月}` を直接実行 → 結果報告
- 集計は既存タブにマージする再実行安全な作りなので、**集計後の遅れ回答も同じ一言で何度でも再集計可**。月指定も可(「8月分の〜」)
- pending に finalize board がある時は board 承認ルート(Mode 5)優先、無い時だけ直接実行(SKILL に明記)
- bot 再起動済(keepalive 経由)。`--output_suffix _test` の安全確認ルートも案内に含めた
