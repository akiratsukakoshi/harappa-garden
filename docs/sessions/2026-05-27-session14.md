## 議論したが結論未定

### 書き戻し経路 / バッドチャンクの掃除

writeback 初版で誤った形式の chunk (`h:e509d3ee` 等、`h:+` プレフィックスなし)を CouchDB に作成。これらは現在 orphan で何も参照していないが、後でクリーンアップしたい。

### Calendar MCP 認証

Google Calendar MCP は `! Needs authentication` のまま。明朝の morning-briefing は `⚠️ カレンダー取得失敗` モードで動く。塚越さんの判断待ち(優先度低)。

### gakuchovault Eden field の本来の用途

`eden: {}` を常に空で push しているが、LiveSync の Eden 機構は小さい doc のインライン格納用らしい。現状は空でも動作 OK だが、本来の使い分けは未調査。

## 達成感

セッション 13 で「Phase 3a A-1 が完全に動いた」と書いたが、塚越さんが「Obsidian に反映されない」と気づいた瞬間に **書き戻し経路の欠如** が顕在化し、Phase 3a は実は **道半ば** だったことが判明。

その場で 2 → 3 を後回しにして書き戻し経路を実装、LiveSync ソースまで降りて chunk ID 仕様を解析、修正版で Obsidian に届くまで完走 → **Phase 3a の本当のラストピース** が嵌まった。

今や Garden は:

1. **書き** = claude -p (種) が VPS で MD を編集
2. **同期** = writeback-daemon が暗号化 + CouchDB push
3. **配信** = LiveSync が PC / iPhone に配信
4. **読み** = 塚越さんが Obsidian で参照
5. **編集** = 塚越さんが Obsidian で書く
6. **逆同期** = LiveSync → CouchDB → mirror-daemon → VPS MD
7. **読み戻し** = 次の種が読む

の循環が **完成**。明朝起きたら recurring-spawn + morning-briefing が cron で動き、その結果が Obsidian で読める ─ という Phase 3a の本来の動作が初めて実証されるはず。

セッション9(`.scratch/` 試作)→ セッション10(LiveSync)→ セッション11(VPS 管理)→ セッション12(mirror-daemon)→ セッション13(ランチャー + ADR 整合)→ **セッション14(書き戻し完成)** で、Phase 3a に必要な全コア要素が出揃った。
