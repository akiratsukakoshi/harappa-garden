# 2026-06-06 セッション36 — Discord 夜/朝の停止を修復(S34 rsync が実行ビットを剥がした)

参加者: ガクチョ(塚越暁)、Claude (Opus 4.8)

> 朝イチの障害対応セッション。「昨晩から Discord の夜の振り返りと朝のブリーフィングが止まった。
> 昨日 LINE にテストを足したのが影響してるのでは」というガクチョの見立てが**ドンピシャ的中**。
> S34 の LINE デプロイ(`rsync -a`)が原因だった。

## 症状

- 6/5 夜から Discord への **夜の振り返り(night-cheer 22:40)** と **朝のブリーフィング(morning-greet 06:40)** が停止。

## 原因(根本原因まで確定)

- VPS cron は起動していたが、ログ末尾に `run-{night-cheer,morning-greet,bot}.sh: Permission denied`。
- **3 スクリプトの実行ビットが剥がれていた**。git 上で `100644`(実行ビット無し)のまま commit されており、
  **S34(6/5 日中)の LINE デプロイ `rsync -avh`(`-a` が perms を伝播)が VPS 側の実行ビットを 644 で上書き**した。
- それ以前は VPS 側に手動付与した実行ビット(Jun 2 20:10)が生きていたため動いていた。最後の成功は
  night-cheer=6/4夜・morning-greet=6/5朝 → **デプロイ後の 6/5 夜 22:40 が初回失敗**。症状「昨晩から」と完全一致。
- `run-send-pending.sh` だけ無事だったのは、これだけ git 上で `100755` だったため。

## 波及

- **`bot.py` 本体は生存**(rsync 前から常駐、keepalive は死活時のみ再起動するため難を逃れた)。
  ただし keepalive(`run-bot.sh`)も Permission denied だったので、**落ちていたら復活不能**な危うい状態だった。

## 対処

1. ✅ VPS で 3 本に `chmod +x`(`run-bot.sh` / `run-morning-greet.sh` / `run-night-cheer.sh`)→ 今夜・明朝・keepalive 回復
2. ✅ 手動 `run-morning-greet.sh` 実行 → 今朝飛んだ分を投稿(`sent for 2026-06-06`、Discord 着弾をガクチョ確認)
3. ✅ **ローカル repo を `644→755` に修正**(根本原因の修正。次回 rsync で再び剥がれない)
4. ✅ `DEPLOY.md` の rsync 手順に注意書き追記(新しい `*.sh` は `git update-index --chmod=+x` で 755 にしてから push)

## 学び

- **`rsync -a` は perms を運ぶ**。デプロイ対象に実行スクリプトがあるなら、repo の mode bit が VPS の正本になる。
  → `*.sh` は git で `100755` を死守。`git ls-files -s` で確認する習慣を DEPLOY 手順に組み込んだ。
- **「最近の変更が引き金では」というガクチョの直感は強い**。S34 の rsync という1点に当たりをつけたのが最短経路だった。
- cron が「起動はするが Permission denied で沈黙」する障害は、**プロセス生存だけ見ると見逃す**(bot.py は生きていた)。
  ログ末尾の `Permission denied` まで読んで初めて分かる。
