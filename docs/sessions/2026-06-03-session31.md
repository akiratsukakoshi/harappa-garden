## 3. B = ガクコ社内チャンネル参加 — 方針の大転換

### 3.1 発見:前提が逆転していた

gaku-co5.0(docker `gaku-co5`、Up 2 weeks 稼働中)は **既に実装済み**:LINE リアルタイム対話(webhook → reply)/ メンション制御(Stage1 `should_respond`、Haiku が会話履歴で判定)/ scope 分離(personal/core_team/staff/aibou/discord)/ scope 別 persona・capability(channels.py)/ scope 記憶(RAW+WIKI+夜間 03:00)/ Anthropic SDK。

→ S22 統合 ADR の前提「garden-gaku-co 本流・gaku-co5.0 レガシー」は、LINE/マルチ scope では**逆転**。Garden 側(memory A〜C)は gaku-co5.0 を別建てで作り直していた。

### 3.2 私の誤り → ガクチョの指摘で軌道修正

私は「案B = gaku-co5.0 を活かし soil/memory を Garden 正本に向けて共有」を推奨。**これは誤り**。ガクチョ指摘:gaku-co5.0 は**社外(デジタル原っぱ大学 / AIBOU LAB)**も相手にする。soil/memory 共有は社外への漏洩経路を作る。スタッフ情報が社外スレッドに流れるリスクは、スタッフ間混線より重大。

→ **gaku-co5.0(社外)と garden-gaku-co(社内)は物理デプロイ分離(エアギャップ)。gaku-co の機能はレガシーとして Garden 側に引き継ぐ**。

### 3.3 過去ログ発見:2026-05-28 ADR 決定4 が既にこれ

ガクチョ「以前の対話ログが残っているはず」→ [2026-05-28 ADR](../decisions/2026-05-28-garden-gaku-co-interaction-layer.md) 決定4(内側/外側エアギャップ)+ 決定6(良い部分を意図的に移植)+ 決定8(段階移行)がまさに同じ結論を明文化済み。私が今回道を誤ったのは、後発 2026-05-31 ADR の「gaku-co5.0 撤退/凍結」+ MAP「統合最終 = シャットオフ」に引っ張られたため(社外専用化=生存 と 撤退=消滅 の食い違い)。

### 3.4 頭脳 = チーム は b(API Haiku)

2026-05-28 決定3 通り。S31 で「なぜ b が安定/安全か」を言語化:
1. 毎メッセージのゲート経済(「いつ黙るか」を ~100token Haiku で安く高速)
2. サブスク同時セッション競合の回避(cron 種 + master bot が claude -p 使用中)
3. LINE reply トークンの時間窓に間に合う速度
4. **情報境界を capability で構造保証**(財務/給与 tool が staff scope の手に存在しない)= prompt 頼みより堅い

master は a(claude subprocess)現状維持。

### 3.5 ベンダーロックイン懸念 → 「skill = SDK tool」の誤解を訂正

ガクチョ「ベンダーロックイン回避が前提。現 skills は Claude Code 依存?」→ 現 skill(SKILL.md)は**プロジェクト内 markdown で中立**(Claude Code Skills 機構不使用、memory 既決)。私の「skill = SDK tool」は雑だった。正しくは 3 層分離:

| 層 | 中身 | ベンダー依存 |
|---|---|---|
| 知識層 | SKILL.md / CHARTER / persona(markdown) | 中立 |
| 行動層 | 素 Python 関数 + 中立スキーマ + capability | 中立 |
| LLM バックエンド | a / b / 将来他社 | **アダプタ 1 枚だけ** |

function-calling は cross-vendor 標準。**「ベンダー中立 = API を使わない」ではなく「アダプタで隔離する」**。b 採用と中立が両立。

## 7. 学び

- **後発 ADR / MAP 要約が前提を誤読させる**:2026-05-31「撤退」+ MAP「シャットオフ」が、2026-05-28 決定4「社外専用化(生存)」を上書きして見え、私を誤誘導。正本 ADR を訂正注記で揃えることの重要性。
- **ベンダー中立の再定義**:「API を使わない」でなく「アダプタ 1 枚で LLM を隔離し、知識層・行動層・capability をプロジェクト内資産にする」。むしろ b(自前 tool)は a(Claude Code 内蔵 tool)より移植耐性が高い。
- **最小実スライス + smoke test**:中立基盤を抽象フレームワークとして空中設計せず、MockProvider で配線だけ接地させてから機能を積む(過剰設計回避、2026-05-28 決定6 と整合)。
- **エアギャップは信頼境界で引く**:「gaku-co か Garden か」でなく「社内(Garden 接続可)vs 社外(隔離)」。社外混線リスクがスタッフ間混線より重大、というガクチョの優先順位が設計を確定させた。
