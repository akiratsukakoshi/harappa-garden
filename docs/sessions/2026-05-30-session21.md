## 4. post_approval 経路 = send_pending.py(セッション選択 (4) C 案)

ガクチョと議論した3案(A: 専用 daemon / B: bot.py 統合 / C: cron 1分毎)から **C 採用**:
- 「月1回の配信に秒単位の即応性は不要」「既存プロセスを汚さない」「bot 落ちと無関係」

### [garden/services/garden-gaku-co/send_pending.py](../../garden/services/garden-gaku-co/send_pending.py) 仕様

- cron `* * * * *` 起動
- `garden/board/pending/*.md` を scan
- frontmatter `status` を検出:
  - `status: test` → ディスパッチ先を **personal** に強制(ガクチョ個人 LINE)→ 配信後に status を `pending` に戻す + 末尾に `<!-- test_sent_at: ... -->` 履歴追記
  - `status: approved` → 通常ディスパッチ。`scheduled_send` が未来なら静かに skip(次の cron 起動で再評価)
- frontmatter `from_seed` でディスパッチ:
  - `shift_manager/monthly-shift-survey` → `/api/send` (staff) → `/api/approve/{id}` (staff は強制承認制のため 2 段呼び)
  - `shift_manager/monthly-working-hours-confirmation` → 同上
  - `shift_manager/month-end-working-hours-prep` → shell 実行(`frontmatter.execute_command`)
- 成功 → board を processed/ へ移動 + Discord master 通知
- 失敗 → pending 残置 + Discord master 通知(エラー詳細)

## 5. Mode 3 「稼働の見せ方」 = URL 方式

### 議論の経緯

候補3案を比較:
- (a) 個別テキスト要約: スタッフ LINE 個人 ID 収集 + bot 友だち追加が必要(準備重)
- (b) Google スプシ個人タブ + 本人だけ閲覧権限: メアド収集 + 権限自動管理(準備重)

ガクチョ「(c) でいきたいけど、スクショは実装できる?」 → 検討:
- LINE Bot API は **PDF (file) を直接送れない**(画像/動画/音声のみ対応)

- スプシ URL 共有が **「過去月比較」「自分のタブ表示」など、スクショより便利な側面** あり

→ **URL 方式採用**(中長期で (a) 個別送信に進化検討)

### workflow との整合性確認

ガクチョの「ワークフローを確認してくれている? HMC のみかな?」指摘を受けて [monthly-cycle.md](../../garden/soil/workflows/monthly-cycle.md) を再読。「(b) 前月稼働の確認依頼送信」が monthly-working-hours-confirmation 種に対応、URL 方式は workflow の改善余地表「(b) スタッフへの見せ方」の候補から選択。**workflow が正本ポリシーを守った設計**。

## 7. テスト配信機能(status:test → personal LINE)

ガクチョ「メッセージの内容を事前に確認したいのと、出来ればテストを事前に行いたい(配信について)」

設計:
- board のチェックリストに「テスト配信 / 本配信 / 却下」の 3 つを並べる
- 庭師が `status: test` に変更 → 保存 → 1 分以内に personal LINE に配信
- 配信後 status は **自動で pending に戻る**(同じ board でテスト→修正→テスト の反復可)
- 本配信は `status: approved`(scheduled_send 時刻に staff へ)

実装は send_pending.py の dispatch ロジック分岐 + reset_test_status() で対応。

## 8. コドモン CSV 自動取込

### ガクチョの要件

「コドモンのCSVからの表への反映はAIにお願いしたい。調べた感じだとAPI,MCPの公開をコドモンはしていないのでcsvファイルは私が指定の場所に置くのでそれを読み込み、該当者に反映する、という風にしてほしい」

### 提供されたサンプル + 実体構造

CSV 列(コドモン「職員入退室エクスポート」):
- 職員コード, 氏名, グループ, 職員, 所属, 日付(YYYY-MM-DD), 曜日, シフト, シフト出勤, シフト退勤, 出勤, 休憩, 戻り, 退勤, 業務開始, 業務終了, **業務時間(HH:MM)**, 備考
- エンコーディング: **Shift-JIS**(コドモン書き出し既定)
- 1 行 = 1 スタッフ × 1 日

### 実装

- [garden/services/shift-manager/import_kodomon.py](../../garden/services/shift-manager/import_kodomon.py)(新規 230 行):
  - CSV を Shift-JIS で読み込む(失敗時 cp932 / utf-8 にフォールバック)
  - `{正式名称: {日付: 業務時間}}` 辞書を構築
  - `DB_Master_Nicknames` シートで氏名照合(空白除去で正規化)
  - working_hours sheet の `YYYY-MM_稼働時間` タブの **放サボ列**(`cat_label_row` から判定)× スタッフ行 を割り出し
  - 該当セルに業務時間(`HH:MM` 形式)を batch_update
- [garden/services/shift-manager/run_month_end_collect.sh](../../garden/services/shift-manager/run_month_end_collect.sh)(新規):
  - 引数 `$1 = YYYY-MM` で:
    1. `generate_working_hours.py --month $MONTH` で稼働シートタブ生成
    2. `garden-mirror/garden/inbox/kodomon/$MONTH.csv` があれば `import_kodomon.py` で放サボ反映
    3. CSV なければ警告のみで続行(放サボ列は手入力のまま)
- month-end-working-hours-prep 種の `execute_command` を `run_month_end_collect.sh` に切り替え

### CSV 配置パス

`garden/inbox/kodomon/{YYYY-MM}.csv` を新設([README](../../garden/inbox/kodomon/README.md))+ `.gitignore` で `*.csv` を除外(個人情報)。

## 10. NPM 経由 `bot.harappa.monster/api/send` 認証懸念

- (案 X) NPM 側で IP allowlist
- (案 Y) gaku-co5.0 `/api/send` に API key 認証追加
