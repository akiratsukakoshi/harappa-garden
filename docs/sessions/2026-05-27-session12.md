## 今日決めたこと(主要)

→ ADR 全文: [docs/decisions/2026-05-27-mirror-daemon-implementation.md](../decisions/2026-05-27-mirror-daemon-implementation.md)

| 決定 | 一行 |
|---|---|
| **スコープ = 単方向**(CouchDB → MD) | 書き戻しは Phase 3a A-1 + 連絡板設計と同期。merge 競合を避ける |
| **実装 = Node.js + octagonal-wheels 直接利用** | 自前実装は仕様追従コストが高い。npm install で本家の暗号モジュール |
| **PBKDF2 salt は `_local/obsidian_livesync_sync_parameters` doc に格納** | LiveSync 内部仕様の発見。passphrase だけでは復号不能、salt も必須 |
| **配置 = VPS `/home/vps-harappa/garden-mirror/`** | 種は VPS で動くので一致 |
| **スコープ = MD のみ** | 添付・画像は当面対象外(種は MD しか読まない) |
| **chunk キャッシュは in-memory LRU(5000 件)** | content-addressed なので再利用可。再起動でリセット(復旧コストとシンプル化のトレード) |
| **state.json で last_seq + path_by_id 永続化** | 再起動で中断地点復帰。削除イベントで path 引きに必要 |
| **Garden サービス追加パターンの 2 例目** | `garden-couchdb_default` external network 参加(NPM は触らない、WAN 公開不要のため) |

## 議論したが結論未定

### 書き戻し経路

- 単方向で始めたが、種が MD を書きたいケース(連絡板への配信本文書き込み、`[x]` archive 転記など)が来る
- Phase 3a A-1 本番ランチャー + 連絡板設計と同期して、書き戻し経路をどこに置くか判断する(次セッション以降)

### chunk キャッシュ永続化

- 現状 in-memory のみ → 再起動で再 fetch / 再復号
- 大量変更時に遅延の原因になる可能性あり。Phase 3a 後半で運用しながら判断

### バイナリ対応(添付ファイル)

- 種は当面 MD のみ。将来必要になれば `doc.path` 拡張子判定を緩めて対応
- LiveSync 側で chunk が同じ E2EE スキームに従うので、daemon 改修は軽量
