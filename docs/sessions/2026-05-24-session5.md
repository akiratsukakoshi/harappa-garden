## 今日決めたこと(詳細は ADR)

### 決定1: workflows/ が正本(SKILL / 他データソースは追従)

HMC SKILL の記述・Google Sheet の構造・運用ドキュメントなどとずれた場合、**workflows/ が勝つ**。発見したら他を直す。

### 決定2: workflow は「目的が正・方法は改善対象」(A 案テンプレ)

各ステップを以下の三層で記述:

- **目的(Purpose)** — 不変
- **現状の方法(Current Method)** — 変更可能
- **改善余地(Improvement Hints)** — ❓未検証 / 💡着手可能 / ✋検討済 / 🛠️実装中 のマーカー付き表

### 決定3: 規律を CLAUDE.md と workflows/README.md に明文化

Claude がセッション開始時に必ず読む位置で「目的が正・方法は改善対象」原則を周知。

### 決定4: 種は責務で分割(同タイミングでも分ける)

月初1日に同時発火する「アンケート送信」と「稼働確認依頼」も別の種にする。理由は責務・失敗スコープ・ライフサイクルが独立しているため。

### 決定5: 最初の種 = `shift_manager/monthly-shift-survey`

セッション4 で挙げていた「月次シート生成」は workflow 解釈の誤りで却下。代わりに月初1日の翌々月アンケート送信を最初の種に。理由: ガクコ統合(/send approval=true)・剪定フロー(LINE personal)・cron 種 の3つを一度に実証できる試金石。

[docs/decisions/2026-05-24-workflows-as-truth-and-improvement-targets.md](../decisions/2026-05-24-workflows-as-truth-and-improvement-targets.md) に決定1-3 をまとめた。決定4-5 は本セッションサマリと MAP.md に記録。

## 種スキーマの位置づけ(セッション中の整理 — YAML 化前の合意)

- **種** = 庭の中で AI を自律発火させる引き金の単位
- **YAML** = その引き金の定義書(single source of truth)
- **表現する**: いつ点火 / 何を実行 / 結果の置き場 / 剪定依頼 / メタデータ / 失敗時
- **表現しない**: 業務全体フロー(→ workflows/) / 実行ロジックの中身(→ コード/SKILL) / 業務知識(→ soil/)
- **性格**: 「点火と連絡先だけの薄いラッパー」
- **安全弁**: `status: active` にならない限り発火しない(YAML を置く ≠ 動き出す)
- **3層関係**: workflows/(業務地図) → garden/seeds/(点火スイッチ) → HMC コード/SKILL(実行の中身) → garden/board/(剪定待ち) → ガクコ(出口)

## workflows/monthly-cycle.md の構造改訂

**旧構造**: 月初1日 / 10日 / 適宜 / 月末 の4ステップ、目的記述なし
**新構造(A案)**: 月末 / 月初1日 / 月初10日 / 適宜 の4ステップ、各ステップに目的・現状の方法・改善余地

時系列順を「月末から」に並べ直したのは、月末稼働表が月初1日確認依頼の前提になる依存関係を見える化するため。

主な追記:
- 月初1日に「前月稼働の確認依頼」(3ルート精算: 業務委託請求書 / アルバイト freee 精算 / 外部スタッフ LINE 申請)を明記
- 月末ステップを詳細化(monthly シート検証 → generate_working_hours → コドモン手入力)
- 各ステップに改善余地表(計15項目程度)
