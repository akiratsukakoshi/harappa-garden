### O2: 番人(Watcher)新設 — cron 失敗の沈黙を破る ⭐本セッションの主役
- [`garden/services/watcher/`](../../garden/services/watcher/) 新設(log_watcher.py + run-watcher.sh + README)
- **エラースキャン**: `garden/log/*.log` の追記分だけをオフセット管理で読み、エラー行を Discord master に通知
- **ハートビート**: `run-send-pending.sh`(毎分)/ `run-bot.sh`(2分毎)が touch するマーカー
  `.heartbeat-*` の鮮度で「cron 自体の沈黙」(S36 型)を検出。警報は 6h に 1 回に抑制
- 実装中の学び 2 つ:
  1. **ログ mtime は生存確認に使えない**(idle 時は書かれない)→ touch マーカー方式に転換
  2. **urllib デフォルト UA は Discord が 403 で弾く** → send_pending.py と同じ User-Agent を付与
- セルフテスト GREEN(エラー行注入 → Discord 通知着弾)。cron `*/10` 登録済み
- Phase 2 ロードマップ「VPS 信頼性 watcher」の前倒し消化

### C1/C2: freee client 統一 + 堅牢化
- **正本** [`garden/lib/freee_client.py`](../../garden/lib/freee_client.py) 新設。
  service は VPS 個別 rsync の自己完結構造のため「正本 + 機械コピー」方式
  ([`sync-freee-client.sh`](../../garden/lib/sync-freee-client.sh) で配布、pre-commit でドリフト検査)
- 強化 3 点:
  1. **429/5xx 指数バックオフ自動リトライ**(Retry-After 尊重)
  2. **refresh 前の token ファイル再読込** — expense と shift が物理 token を共有しており、
     片方が refresh すると他方の in-memory refresh_token が失効する競合の対策(潜在バグ修正)
  3. account_items / sections / taxes / partners の**インスタンスキャッシュ**
- VPS 実証: expense `processor.py taxes` で **401 → refresh → リトライが実走 GREEN**(tax_code 136 確認)、
  shift-manager 側も get_taxes + キャッシュヒット GREEN
- ガクチョ判断により expense「本番 1 周待ち」凍結を解除して先行実施(月末テストがリファクタの
  初検証にならないよう、read-only 実証まで本セッションで完了)
