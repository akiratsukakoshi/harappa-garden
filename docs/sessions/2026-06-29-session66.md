## 実施5: ガクコの空返事対策

`garden/services/garden-gaku-co/bot.py`:

- Discord master の prompt に「空返事禁止・実行確認ルール」を追加
- 「反映した / 作成した / 予約した / 送った / 進めます」と言う前に、board / log / stdout / 外部 API 結果など完了根拠を確認するよう明記
- 完了根拠がない時は「まだ動いていない」「ここで止まっている」「権限/入力が足りない」と返すよう明記
- cron 窓を過ぎた workflow では、次回 cron 待ちにせず手動起動が必要か判断するよう明記
- SNS 後追い承認では、月曜 07:30 を過ぎていたら「月曜朝に走ります」と返さず、`monday-caption-draft` を手動起動するか、できない理由を返すよう明記

`garden/services/garden-gaku-co/line/context.py`:

- LINE core_team の system prompt に同じ空返事禁止ルールを追加
- tool 失敗、対象不在、外部サービス到達不能、次段自動実行条件未充足時は完了したふりをしないよう明記

`garden/services/garden-gaku-co/brain/respond.py`:

## 入力トークン確認

ガクチョから system prompt 肥大の懸念があり、VPS 実コードで概算した。

Discord:

- 通常会話: 約 29,700 chars / 約 7,400〜9,900 tokens
- SNS 話題: 約 36,700 chars / 約 9,200〜12,200 tokens
- 経費話題: 約 43,100 chars / 約 10,800〜14,400 tokens
- 財務話題: 約 44,300 chars / 約 11,100〜14,800 tokens

LINE:

- core_team サンプル: 約 11,000 chars / 約 2,800〜3,700 tokens

今回追加した空返事禁止ルール自体は数百文字で、肥大の主因ではない。主因は Discord 側の CHARTER / daily-pilot SKILL / memory wiki + RAW / topic SKILL block / pending board / 会話履歴。

## 宿題

- **ガクコ空返事対策の構造化**: prompt ルールだけでなく、実行系 tool / deterministic route に「完了根拠」「未完了理由」「次段起動要否」を返す共通 envelope を導入する。
- **SNS 後追い承認の自動化**: `sns-select` が月曜 07:30 後に approved になった場合、`monday-caption-draft` を自動起動する event route を検討する。ただし重複生成ガードが必要。
- **SNS caption 承認→予約の構造化**: `sns-caption` board 承認時に、火単写真 / 土カルーセルの caption file 生成と `processor.py schedule` 実行を確実に行う route を点検する。
- **Discord prompt 圧縮**: `MEMORY_MAX_CHARS=30000` と各 topic SKILL block の読み込みを見直す。通常会話では memory summary のみ、topic 検知時だけ該当 SKILL の要約または必要 section を load する方向。
- **LINE prompt / tool schema 圧縮**: 現状は許容範囲だが、RAW_CHARS_CAP=20000 のため長期的には scope memory の要約化が必要。
- **launcher 通知のテスト追加**: `==NOTIFY==` dispatch の unit test を追加する。
- **VPS 余分ファイル整理**: 一度の rsync 誤指定で `/home/vps-harappa/garden/services/garden-gaku-co/context.py` / `respond.py` が直下にできた。import には影響しないが、次回安全に削除する。
- **Zoom orphan 確認**: 初回固定会議 CLI 失敗時に orphan Zoom ができた可能性がある。Zoom list/read scope を追加するか、管理画面で確認する。
