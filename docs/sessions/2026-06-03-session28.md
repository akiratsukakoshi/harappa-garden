
朝のブリーフィングは正常だったとのこと。

## 1. 原因特定 — 取りこぼし 2 段階

### 1.1 night_cheer.py の path 計算が旧パスのまま

[night_cheer.py:51](../../garden/services/garden-gaku-co/night_cheer.py#L51) の `read_review_log`:

```python
path = os.path.join(MIRROR_DIR, "garden", "log", f"{d.isoformat()}-night-review.log")
```

= `/home/vps-harappa/garden-mirror/garden/log/...` = **vault 内の旧パス**。S27 で `send_pending.py` / `launcher.mjs` / `run-*.sh` の env default は新パスに切り替えたが、night_cheer.py は対象から漏れていた。

VPS 上の実ファイル確認: `/home/vps-harappa/garden/log/2026-06-02-night-review.log`(4178 bytes、22:33 完了)は **新パスに存在していた**(night-review 本体は正常完走)。

### 1.2 settings.json に log の Write/Edit 権限が無い

night-review の prompt は「ログに `==NOTIFY==..==END==` ブロックを **Edit で append** しろ」と指示している。これは Claude が log ファイルに書き込む前提。しかし VPS の `~/.claude/settings.json` の `permissions.allow` に:

- `Write/Edit(//home/vps-harappa/garden-mirror/garden/**)` ← 旧 log パスはカバー
- `Write/Edit(//home/vps-harappa/garden/board/**)` ← S27 で追加
- **`Write/Edit(//home/vps-harappa/garden/log/**)` ← 無い**

実際の night-review log を読むと claude 出力に:
```
`/home/vps-harappa/garden/log/` への書き込みに権限が必要です。承認をお待ちしています。
```

= **NOTIFY ブロックを書けず log に残らなかった**。だから night_cheer.py は仮に新パスを見ても件数が取れない構造。

つまり「path 取りこぼし」+「権限取りこぼし」の **2 段階失敗** が同時発生していた。

## 2. 修正実施

### 2.1 night_cheer.py(repo + VPS)

[night_cheer.py](../../garden/services/garden-gaku-co/night_cheer.py) に `LOG_DIR` 環境変数を追加し、`read_review_log` のパス計算を切り替え:

```python
MIRROR_DIR = os.environ.get("MIRROR_DIR", "/home/vps-harappa/garden-mirror")
LOG_DIR = os.environ.get("GARDEN_LOG_DIR", "/home/vps-harappa/garden/log")  # 追加
...
def read_review_log(d):
    path = os.path.join(LOG_DIR, f"{d.isoformat()}-night-review.log")  # 切替
    ...
```

`MIRROR_DIR` 自体は `archive.md`(vault 内 `hmc_tasks/`)で引き続き使うので残す。log だけ env を分離する設計。scp で VPS の `/home/vps-harappa/garden/services/garden-gaku-co/night_cheer.py` に反映。

### 2.2 VPS settings.json

`~/.claude/settings.json` の `permissions.allow` に 2 行追加(backup `~/.claude/settings.json.bak.YYYYMMDD_HHMMSS` 取得後に Python で json 直接編集):

```json
"Write(//home/vps-harappa/garden/log/**)",
"Edit(//home/vps-harappa/garden/log/**)"
```

### 2.3 副作用ゼロの権限検証

軽量 write テスト:
```
claude -p 'Use the Write tool to create the file /home/vps-harappa/garden/log/_perm-test.log with the single line content: perm-check-ok' --model haiku
```

→ ファイル生成成功 → 削除して片付け。Discord ノイズゼロで権限確認完了。

## 3. 検証時の小ミスと対処

最初に「path 修正の動作確認」と称して `run-night-cheer.sh` を朝 9:38 に手動実行した。これは設計上 `today_jst()` = 今日(2026-06-03)の log を探すロジックで、今日のは未来(まだ無い)→ 「⚠️ 見つかりません」警告がもう 1 通 Discord に飛んだ可能性が高い。

学び:**cron 起動を想定した bottom-up スクリプトを手動で叩く前に、内部の "今日依存ロジック" を確認すべき**。

切り替え後は副作用ゼロの方法に変更 — Python から関数を直接呼んで日付指定で path 解決のみ確認:

```python

d = datetime.date(2026, 6, 2)
log_text, path = night_cheer.read_review_log(d)
# path resolved: /home/vps-harappa/garden/log/2026-06-02-night-review.log
```

## 6. 学び

- **path 大移動の後始末はチェックリスト化したほうがいい** — S27 で `send_pending.py` / `launcher.mjs` / `run-*.sh` / 種ファイル / OPERATIONS.md / MAP.md は更新したが、**night_cheer.py(夜の声がけ)と settings.json(claude 権限)が漏れた**。次回同等の path 移動が起きるときの教訓
- **cron 想定スクリプトの手動実行は罠** — `today_jst()` のような "今日依存" ロジックは、cron 時刻と手動実行時刻で振る舞いが変わる。手動デバッグの前に内部ロジックを確認、または関数を直接呼ぶ副作用ゼロの検証に切り替える
- **副次発見の価値** — 1 つの bug を追っていたら同じ構造の別 bug が見つかった(ingest-raw の permission も同じ取りこぼし)。さらに ingest-raw のコア処理は成功していたので、**Stage A.5 が想像していたより早く効果を出していた** ことも分かった
- **小さな修正でも backup を取る** — settings.json は手で書き換える前に必ず `cp ... .bak.$(date)` で退避。コストは無視できるレベル
