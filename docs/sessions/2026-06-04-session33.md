# 2026-06-04 セッション33 — 業務 Garden 化の型(plot_gardener)+ 語彙 2 register 整理

参加者: ガクチョ(塚越暁)、Claude (Opus 4.8)

> 出先セッション。測量士の手紙(2026-06-03)を起点に、「業務を Garden 化する型」を確立した。先にガクチョと **語彙を揃え**(これが本セッションの肝)、そのうえで測量士の 5 提案をすべて採用 → メタ区画 `plot_gardener` をフル本番化。VPS 反映は不要(repo 内ドキュメント + SKILL のみ、soil/memory 未編集)。実装(dogfood)は次セッションへ。

## 0. 文脈

S32 の申し送り「(次フェーズ)core_team 業務 tool 整備 = 測量士の plot_gardener 手紙を参照して計画」を消化するセッション。ガクチョの当初の問いは「ツール類の実装計画について相談したい」。

## 1. 測量士の手紙の肝(と問いの再構成)

測量士の主張:**個別 tool を量産するな。業務フロー → 区画への変換の「型」(=メタ SKILL `plot_gardener`)を先に作れ。** tool の粒度は Garden が決めるので、ガクチョは「この業務を Garden 化して」と業務単位で渡すだけにする。

ガクチョの確認:「測量士と私の認識は合っている。あの手紙は私とのやり取りで作ったもの」。→ tool 単体には意味がない。ワークフロー起点の **種 + SKILL + 通行手形** の選択が重要。shift_manager と daily-pilot で既に実証済。同じことを効率的に量産したい、が真のニーズ。

## 2. 語彙合意(本セッションの中心)

ガクチョの問い 2 つに答える形で定義を確定:

### Q1: 「業務のパック」= **区画(Plot)**

ガクチョの言う「ワークフロー起点の 種 + SKILL + 通行手形 + service の束」= Garden 語彙の **区画(Plot)**。shift_manager / daily-pilot がその第 1・2 号。`plot_gardener` は「区画を作るメタ区画」。→ **ガクチョが「区画の意味を正確に理解していなかっただけ」だと自覚 → すっきり解決。**

### Q2: 「tool」は **使う。ただし水面下の実装層用語。Garden 語彙表には載せない**

- Garden 語彙 = 比喩 = 庭師の思考言語。tool は比喩じゃない(ただの実装単位)。
- だが消せない(通行手形が「tool 名の集合」で定義される以上、code/S31 ADR の土台)。
- → **2 register に分離**:設計言語(区画・種・通行手形=庭師が設計)/ 実装層(tool・service=Garden が選ぶ水面下)。

### おまけ: capability → **通行手形** に昇格

語彙表「今後追加候補」に既にあった「権限境界/通行手形」を本採用。コード内は英語 `capability` 維持、議論では「通行手形」。

→ **ガクチョ「全て合意」**。

## 4. 設計の到達点

- 庭師が設計するのは **区画・種・通行手形** の 3 つだけ。**tool/service は水面下で Garden が選ぶ**。
- ガクチョが渡す依頼単位 = **業務名 / mode / MVP** の 3 つ。
- 移植型(transplant)は **Legacy Inventory から**。合言葉「業務知識は継承、起動と承認だけ Garden に変える」。
- plot_gardener の価値は「自動化」より「**毎回の判断を議論から分類に落とす**」こと。

## 学び

- **問いの立て方そのものを測量士が再構成していた**:ガクチョの「ツール実装計画」を「業務 Garden 化の型」へ。tool 一覧ではなく型を作る、が正しい対象だった。
- **語彙を先に揃えると、その後の実装が分類作業に落ちる**:「業務のパック=区画」が腑に落ちた瞬間、残りは全部その派生として一気に決まった。plot_gardener 自身の思想(議論→分類)を、本セッションが体現した。
- **比喩の register を分けると tool が暴れない**:tool/service を「水面下」に押し込むことで、ガクチョの認知から実装詳細を構造的に外せた。
