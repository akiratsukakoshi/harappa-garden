## 1. scaffold を作って、京急で捨てた(設計判断の核心)

「真の一発 bootstrap」= ドメインを渡すと soil 台帳ドラフトを自動生成、を狙って `sweep_client.py` に **scaffold モード**(企業 README スケルトン + pack JSON + 担当者 stub を自動書き出し)を実装。MTI/パナHM では綺麗に通った。

→ 3社目=**京急(`keikyu-group.jp`・幅広く事業をする会社)**で実走 → 機械抽出の限界が3つ露呈:
- **漢字氏名が取れない**(ローマ字 local part)→ people stub が `asami.ikeda.md` でブサイク=手戻り増
- **最大クラスタがノイズ**(PPAP 解凍パスワード通知22件)
- **1案件が表記ゆれで20+に割れ + 他社混在**(ゴンチャ/ヤマハ/三浦観光バス)

ガクチョの問い「Python を入れるメリットは? 10-15社の有限・判断主体なら、手動 + SKILL 記述の方が大事では?」→ **同意**。

**方針転換([ADR 2026-06-18](../decisions/2026-06-18-client-bootstrap-manual-not-scaffold.md))**:
- soil 台帳は **Claude が手で起こす**(機械生成は個性を潰し抽出ミスが手戻りを生む)。
- Python(`sweep_client`)は **「Claude が Gmail を直接読めない」を埋める素材ダンプ(`--domain` digest)に限定**。ファイル生成(scaffold/pack/stub)は撤去。
- 力点は **SKILL の「各社の型ライブラリ」**(横展開のたびに追記=仕組みでなく記述が増える)。
- weekly **sweep**(差分監視・cron)は残す(監視であって生成でない)。
- **残した**:案件 frontmatter の枠3つ(finance_links/roles/uncertainties・測量士 #4#5#2)+ パナHM 実当て。測量士 P2④⑤ + pagination テスト。

## 2. 京急を手起こしで完走(横展開3号・型の実証)

最難物の京急で「手動 + SKILL 型」が回ることを実証:

- **素材ダンプ** = `sweep_client --domain keikyu-group.jp`(66スレッド)→ Claude が生件名を読む。
- **クラスタリング**(ガクチョ補完で確定):
  - メイン = **みうらの森林共創パートナー事業**(継続・横須賀市野比の京急所有地。原っぱ=事務局、パートナー企業が月額)
  - 派生3本 = **桐畑(プライム横浜・契約済150万)** / 三浦海岸(イベント提供のみ) / 品川再開発(芽・6/15 Plaud 会議録)
  - 除外:野比かがみ田緑地(別NPO管理の別場所) / ゴンチャ社員総会・茶畑(ゴンチャ直接案件=将来別クライアント)
- **漢字氏名** = 署名本文の inline ダンプ → **苗字10名カンペキ**(ガクチョ確認)。杉山(共創室キーパーソン)・藤吉ほか。個別ページは乱造せず担当者表に集約。
- **桐畑の契約書(PDF)** から finance_links 枠を**実額充填**(70/40/40万・〜2029/3・支払期日)。
- **★商流が逆**(共創パートナー型の特異点・ガクチョ補完):パートナー企業→原っぱ入金(売上)/ 原っぱ→**京急へ外注費支払い**(京急が請求書発行)。京急は「請求先」でなく**支払先**。frontmatter `freee_partner_role: 支払先` を立て finance に伝達。毎月の[共有レポート Sheets](https://docs.google.com/spreadsheets/d/19IWqq-DcBtM8-rOuXxX3xRrqmUzjhcnh5ycjRERtEIY/edit) も記載。

**型ライブラリ(SKILL)に3型確立**:研修連鎖型(MTI)/ 継続運営型(パナHM)/ **共創パートナー型(京急)**。診る普遍ルール4点(主軸見極め / 他社混在の三分 / お金の向き確認 / 署名ダンプで苗字確定)も明文化。

## 今後の流れ(ガクチョの問い:クライアント登録 vs 型整備 への回答)

**結論 = クライアント登録を優先**(型は3社=3型で回ることを実証済み。これ以上の抽象的な型整備は机上。記述=各社の正本が増えることが資産)。

1. **残り主要クライアントを1社ずつ登録**(ゴンチャ / boundlesslife / 白井松 / 三井 ほか):
   `sweep_client --domain {ドメイン}` で素材ダンプ → Claude が手起こし(MTI/パナHM/京急を型参照)→ 漢字取れなければ署名 inline ダンプ → ガクチョ剪定(案件切り分け・金額・確度・ロール・**お金の向き**)→ soil。
2. **登録のたびに型を太らせる**(新しい型が出たら SKILL 型ライブラリに追記。schema が足りなければ枠を足す=安い)。
3. **5社前後で凍結判断**:schema 確定 + 抽出 polish(PPAP 除外・署名漢字抽出の `sweep_client` 正式化)を費用対効果で判断 + events ledger(測量士 #1)が要るか実データで判断。
4. **並行**:既存登録済(MTI/パナHM/京急)の剪定の詰め(京急の担当ロール確認 / 各社の freee 突合)+ weekly sweep が差分監視。
