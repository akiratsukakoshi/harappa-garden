## ★ ガクチョとの認識すり合わせ(移行戦略の確定)

提案4(scribe)の考察から、ガクチョの指摘2点で私の見立てを訂正:

1. **②(`claude -p`)がサブプロセスなのはコスト戦略**(`CLAUDE_CODE_OAUTH_TOKEN`=サブスク認証で定額・API は従量)。ロックインではなく**意図的に置換可能**な構造。
   - 系: **②を純API化するのはコスト逆行**(定額の重い仕事を従量に移す)。移行先は別の**サブスク制エージェントCLI**(Codex/Gemini CLI)であって metered API ではない。runner 抽象がその受け皿。
2. **MCP は共通規格**(Codex/Gemini も読める)。`.mcp.json` の Plaud は **stdio + サーバが refresh 自己所有** → どのエージェントCLIでも透過到達。

**Garden 全体の Claude 到達マップ(grep で確定)**:

| 軸 | 到達手段 | 認証 | 使用箇所 | lock-in |
|---|---|---|---|---|
| ① API経由 | `anthropic` SDK → Messages API | `ANTHROPIC_API_KEY`(従量) | brain/provider・gate・respond・**line/app.py** | 弱(Provider 足すだけ) |
| ② Claude Code CLI | `claude -p` | `CLAUDE_CODE_OAUTH_TOKEN`(サブスク) | **launcher 全 seed**・bot.py・morning/night・scribe・writeback | 強だが置換可 |
| ③ Agent SDK | — | — | **使っていない** | — |

**結論**: 残る本当の依存は transport(MCP/サブプロセス)ではなく **「各エージェントCLIの 引数 / ツール意味論 / 権限形式」の3点**。これは runner(引数)+ grants.yml(権限)が吸収するよう既に足場ができている。**ツール意味論(Write/Edit/Bash の名・挙動)だけは実走しないと差分が見えない**=次の一手。

## 提案4 の再定義(格下げ)

当初「scribe = 最難所(MCP が Claude 固有)」は**誤り**。MCP 共通規格 + サーバ側 refresh で **scribe はただの engine 引数翻訳ケース**。提案4 は engine 移行には不要となり、残る実利は「Plaud 取得を LLM ターンから外す」**コスト/堅牢性の最適化**に降格。やるなら gating(Python stdio MCP クライアントで Plaud を叩けるか)を先に実証。優先度は中以下。
