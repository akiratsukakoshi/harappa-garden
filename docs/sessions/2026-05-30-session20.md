## 3. 論点3: SKILL ↔ soil 参照規約 + 菌糸(Mycelium) 新設

### 重要な見落とし → 統合へ
当初「`garden/soil/INDEX.md` を新設」と提案したが、ガクチョ指摘:「soil は LLM Wiki の考えに則っていて、既に INDEX があるが、これと同位置づけを整理する? 統合するイメージ?」

→ 既存 [garden/soil/index.md](../../garden/soil/index.md) が **Karpathy LLM Wiki 方式の中核** で、まさに私が新設しようとした INDEX と同位置づけ。新設は明確に重複 → **統合採用**。

### 菌糸(Mycelium)の発見
ガクチョ:「インデックスの更新をする(土を耕したり整理したりする)役割。**土壌の菌糸とかかな**? LLM wiki の考えもそこがキモだと思う」

→ LLM Wiki の核心は INDEX の存在ではなく、INDEX を active に維持する役割の存在。soil/README に書かれた3責務(Ingest / Query / Lint)を担うエージェント実体が未定義だった。

**Garden 語彙に「菌糸(Mycelium)」を正式追加**。役割マップ:

| 役割 | 振る舞いの本質 |
|---|---|
| 庭師(Gardener) | 意志を持つ。決める |
| 番人(Watcher) | 監視する。告げる(目) |
| **菌糸(Mycelium)** ← NEW | 分解する。運ぶ。整える(土の中の網目、見えない基盤) |
| 草木 / 木の精 | 表に出て話す(対話層) |

番人と菌糸の違い:番人は「見て告げる」、菌糸は「分解して運ぶ」。責務が違うので独立した役割。

### 菌糸の責務(Mode)

| Mode | 内容 | 発火源 |
|---|---|---|
| Mode 1: Ingest | 新ソース → soil 反映 + index 追記 + log | watcher / 種(cron) |
| Mode 2: Lint | 矛盾・古い記述・孤立 → 連絡板に剪定依頼 + **創発ログ review** | 種(週次 cron) |
| Mode 3: Index 更新 | soil 編集 → index.md 自動追従 | watcher |
| Mode 4: 関係性編み直し | `linked_*` 整備 + `[[link]]` 漏れ検出 | 種(月次) |

### 段階分け(Stage)

| Stage | 何を実装 | タイミング |
|---|---|---|
| **Stage 1** | Mode 3(Index 更新) | **shift_manager より先**(staff 29 名分の index 最新化が前提) |
| Stage 2 | Mode 2(Lint = 週次 cron) | shift_manager 安定後 |
| Stage 3 | Mode 1(Ingest = 議事録・メール対応) | meetings 連携と同期 |
| Stage 4 | Mode 4(関係性 = 月次 cron) | 運用成熟後 |

### 実装(本セッション分)
- [docs/garden-vocabulary.md](../garden-vocabulary.md) に「菌糸(Mycelium)」+ 役割の責務マップ追加
- [garden/CHARTER.md](../../garden/CHARTER.md) に「soil の参照」節を追加
- [garden/mycelium/](../../garden/mycelium/) 新設 + [README.md](../../garden/mycelium/README.md)(位置づけ + Mode 設計 + Stage 分け)
- ADR: [2026-05-30-mycelium-and-soil-reference.md](../decisions/2026-05-30-mycelium-and-soil-reference.md)
- Stage 1 実装は次回着手

当初「INDEX を起動時動的生成(D-γ)」と提案したが、LLM Wiki 哲学(LLM が ingest 時に意味的に更新)に反する → 撤回、手動(LLM 半自動)更新を継続採用。

## 4. 論点4: 創発を SKILL に書き戻すか → 評価駆動

### 課題
S19 で bot が独自に `## 運営・企画(繰越)` セクションを active 冒頭に新設した = **創発**(SKILL 外の動き)。Empowerment & Proactivity 原則からは歓迎、ただし全部書き戻すと SKILL 肥大化、全部黙認すると知識散逸。

### 採用案
**Stage A(今すぐ運用ルール化)+ Stage B(菌糸 Stage 2 Lint と統合)**

**Stage A(CHARTER に運用ルール追記)**:
- 創発はその場限り(一過性)で動く。SKILL を自動更新しない
- 庭師が「これ良い、次もそうして」と評価 → bot が「SKILL に書き戻しますか?」と確認(Pattern A)
- 書き戻す場合は SKILL Edit + log 記録
- 違和感を示したら「SKILL 通りに戻す」で取り消し可能

**Stage B(将来)**:
- bot が「SKILL に書いてない動き」を自己認識 → `garden/board/emergence/{date}-{plot}.md` に記録
- 菌糸 Mode 2(Lint、週次)が emergence ログ review → 繰り返しを SKILL 書き戻し候補としてガクチョに剪定依頼

### 実装(本セッション分)
- CHARTER に「創発(SKILL 外の動き)の扱い」節を追加
- mycelium/README の Mode 2 に「emergence ログ review」を追記
- VPS scp + bot 再起動

## 5. 論点5: 永続記憶 — gaku-co5.0 を移植

### ガクチョの要件
1. 覚えたいもの優先順位:**判断ログ > 背景 > 対話文脈**。グレーな約束は拾わない(明示的タスク化が境界)
2. 「日誌 + 覚書」分離構造は賛成
3. LLM Wiki に統合できないか? SQLite 必要? **ガクコの実装を確認してほしい**
4. 取得タイミングはやりやすい方から
5. **チャネル階層と記憶の境界**(マスター > 運営 > スタッフALL > イベント別)— マスターは全部見渡せる、ただし下位 → マスター流入、マスター → 下位漏出は避けたい

### gaku-co5.0 解析結果 — 既に完成していた

[/home/tukapontas/gaku-co5.0/app/memory/](file:///home/tukapontas/gaku-co5.0/app/memory/) を確認:

- **2 段構造**: RAW(日次 MD・14日保持)→ WIKI(LLM 抽出・統合)
- **スコープ分離** ([scope.py](file:///home/tukapontas/gaku-co5.0/app/memory/scope.py)): MASTER / LINE_CORE_TEAM / LINE_STAFF / discord_{label} / line_aibou_{group_id}
- **夜間バッチ** ([processor.py](file:///home/tukapontas/gaku-co5.0/app/memory/processor.py)): 03:00 JST(APScheduler)、claude-haiku-4-5 で EXTRACT(人物 / 事実 / 決定事項 / 予定)→ CONSOLIDATE(重複まとめ / 矛盾解消 / index 再生成)→ NEW/UPDATE 書き込み → 14 日経過 RAW 削除
- **DB**: SQLite **不使用**、MD のみ
- **index.md**: スコープごとに存在(LLM Wiki 哲学の中核)

→ ガクチョの要件と **約 90% 一致**。再発明不要 = **移植する**。

### 差分(マスター透視権の追加実装)
gaku-co5.0 の MASTER スコープは「LINE personal + Discord master を同一スコープに流す」設計。ガクチョの欲求は別物で「マスターは下位を読める、ただし下位はマスターを読めない」(階層的閲覧権)。bot.py(マスター)起動時に下位スコープの index.md を全部 read、下位 bot は自スコープのみ read(settings.json path-scoped allow)。Stage D で実装。

### 菌糸との責務一致
gaku-co5.0 の夜間バッチが菌糸 Mode 1 Ingest、RAW 削除が Mode 2 Lint、index 更新が Mode 3 に完全対応。**菌糸が soil と memory 両方を維持** する構造で美しく整合する。

### Garden 統合案

**配置**: `garden/memory/{scope}/`(master / line_core_team / line_staff / discord_* / line_aibou_*)
**Garden 語彙**: 「**記憶(Memory)**」(英語表記そのまま、シンプル)。日誌(Diary)等への細分化はしない(RAW/WIKI 2 段で十分整理)
**段階分け**:

| Stage | 何を実装 | タイミング |
|---|---|---|
| A | gaku-co5.0 app/memory/ を移植 + master scope で対話 RAW logging 開始 | 次セッション |
| B | 夜間バッチ起動(03:00 JST、claude-haiku-4-5)+ 古い RAW 削除 | A 安定後 |
| C | bot 起動時に master index + 最近 RAW を context にロード(真の永続記憶) | B 完了後 |
| D | 下位チャネル開放 + マスター透視権実装 | チーム公開時(Phase 2 と同期) |
