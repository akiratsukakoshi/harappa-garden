# 2026-06-02 セッション27 — 放サボ上書き事故 → 構造改革(board/log を vault 外へ)

参加者: ガクチョ(塚越暁)、Claude (Opus 4.7)

> S26 終了の数時間後、ガクチョが 5 月稼働サマリで **放サボ列が全消失** していることを発見。原因調査で Obsidian Self-hosted LiveSync 経由の board 巻き戻しが特定され、それを起点に「承認運用を Discord ガクコ経由に切り替え + board と log を Obsidian vault の外に出す + daemon に EXCLUDE_PREFIXES を入れて構造的隔離」という大きな構造改革を 1 セッションで完走。事故対応からはじまり、ADR 1 本 + ファイル ~30 + daemon 再ビルド + CouchDB 27 件削除 + ダミーテスト完走まで。

## 0. 起点 — 「5 月稼働サマリで放サボの労働時間が消えている」

ガクチョ報告:「稼働時間の 5 月サマリを確認したところ、上書きされており、放課後サボールの労働時間が消えてしまっています。csv から反映するフローを昨日回したがその後、朝にいろんな cron のエラー処理が回った影響だと思う」

調査開始。

## 1. 事故の構造特定(LiveSync 起因の board 巻き戻し)

### 1.1 直接原因の確認

`send-pending.log` のタイムライン:

- **6/1 22:48:19** — `2026-05-31-working-hours-prep.md` が approved → `run_month_end_collect.sh 2026-05` 実行 → 稼働表タブ生成 + (CSV パス検出ミスマッチで)放サボ列はガクチョ手動入力 → board は processed/ へ移動
- **6/2 09:04:36** — VPS の `mirror-daemon` が CouchDB → fs に **5 ファイルを `board/pending/` に PUT**(2026-05-31-working-hours-prep / 2026-06-01-monthly-shift-survey / 2026-06-01-monthly-working-hours-confirmation / `_test-dummy.md` / `dummy-test-s22.md`)
- **6/2 09:05:01** — `send_pending.py` が再検出 → `run_month_end_collect.sh 2026-05` を **再実行**
- **6/2 09:05:19** — `generate_working_hours.py` がタブを再生成 → **放サボ列が全消失**

### 1.2 根本原因

**Obsidian Self-hosted LiveSync** が VPS の `/home/vps-harappa/garden-mirror/` を真実とする 3 コンテナ構成(CouchDB + mirror-daemon + writeback-daemon)で動作中。ガクチョの Obsidian(PC + スマホ)とこの mirror が双方向同期している。

board のライフサイクル(pending → processed)は VPS 上の send_pending.py が実行するが、別端末の Obsidian が **古い「pending にある状態」を保持してオフラインだった** 場合、再オンライン時に CouchDB へ stale state を push し直す。それを mirror-daemon が VPS fs に展開 → send_pending.py が **既に approved になっている古いファイル** を再検出 → 同じ shell コマンドを **重複実行** → 放サボ列の手入力データが消える。

副次的に、S25 で導入した **連続失敗ガード(fail_count + auto-quarantine)も同じ理由で機能していなかった**:LiveSync が fail_count フィールドを毎ティック上書きでリセット → 閾値到達せず Discord に同じ ❌ 通知が毎分流れ続けるスパムの正体。

## 2. 放サボ列の復元(即対応)

CSV `garden/inbox/kodomon/202605.csv` は無事(5/30 22:06 配置の 24KB)。VPS で `import_kodomon.py --month 2026-05 --csv .../202605.csv` を直接実行 → **3 名 × 4 日程 = 12 セル反映**(5/7・5/14・5/21・5/28)→ ガクチョ確認 OK で復元完了。

## 3. 構造的対策の方針合意

事故の発端は LiveSync 経由の board 巻き戻し。私はまず案 A〜D を並べた:

- **案 A**: LiveSync の ignore pattern に `garden/board/**` を追加
- **案 B**: 承認 UI を Obsidian から離す(Discord ガクコ経由に切り替え)
- **案 C**: send_pending に CouchDB rev 整合性チェックを追加
- **案 D**: `generate_working_hours.py` に既存タブガード(idempotency)を追加

ガクチョ判断:「Discord でガクコとのやり取りで承認することはできる?obsidian を見ることはするが、書きかえるよりはガクコに、承認と伝える方が楽。そしてそれが安全ならそうしたい。」

→ **B 採用** + 副案として、D も実装(防衛二段構え)。さらにガクチョから核心の問い:「VPS で完結するならそもそも vault でファイルを管理する必要ある?board は obsidian の外側でディレクトリ管理する方がシンプルでは?」

→ 構造が一段単純化。board だけでなく **log も含めて vault 外配置**(B 案)に拡張。

最終合意プラン:
- board と log を `/home/vps-harappa/garden/` 配下に移動(vault 外)
- 承認運用は Discord ガクコ経由に切り替え(shift_manager SKILL Mode 5 新設)
- daemon に EXCLUDE_PREFIXES を入れて、CouchDB から VPS への巻き戻しを構造的に遮断
- LiveSync ignore は不要(daemon フィルタが代替)

### 7.4 daemon に EXCLUDE_PREFIXES 機構を追加(LiveSync ignore の代替)

ガクチョの Obsidian で LiveSync の ignore 設定 UI を探すも、3 回画面遷移しても発見できず。**設定 UI 探しを諦めて daemon 側で隔離する代替案** に切り替え。

`mirror-daemon` と `writeback-daemon` に `EXCLUDE_PREFIXES` 環境変数を追加:
- mirror-daemon: `syncDoc` の冒頭で `checkPath` が除外プレフィックスにマッチしたら skip(削除も put も無視)
- writeback-daemon: `inScope()` を `scopePrefixes 内 AND excludePrefixes 外` に変更

docker-compose v1.29 + Docker engine v23 の互換問題(`KeyError: 'ContainerConfig'`)が発生したが、`down → build --no-cache → up -d` のサイクルで回避。両 daemon を再起動し、ログで動作確認(`garden/board/` `garden/log/` 系のログが一切出ない = 完全隔離成立)。

### 7.5 VPS 物理移行 + 後片付け

実行手順:
1. 新ディレクトリ作成 + 必要ファイルを選択的にコピー(掃除も兼ね、`_test-dummy.md` は quarantine 行き、placeholder は除外)
2. crontab を `sed s|garden-mirror/garden/log|garden/log|g` で一括更新(15 行)
3. 旧 vault 内 `garden-mirror/garden/{board,log}/` を `rm -rf`
4. **CouchDB から旧 board/log document を bulk delete**(27 件、失敗 0):mirror-daemon コンテナ内 node から CouchDB API を直接叩く

ガクチョのスマホ + PC Obsidian で `garden/board/` `garden/log/` が消えるはずだったが「まだフォルダが残ってる」報告。`writeback-daemon` が EXCLUDE_PREFIXES で「削除」を CouchDB に push しないので、CouchDB の document は VPS 側で消しても残る → Obsidian は CouchDB を真実視するため見た目が消えない構造。

代替策:CouchDB から 27 件を `_bulk_docs` で `_deleted: true` 更新(私が VPS から実行)→ それでも Obsidian 側の表示が更新されないため、最終的にガクチョが PC Obsidian で手動削除 → LiveSync で削除イベント push → スマホも追随。`/home/vps-harappa/garden/` 配下の本物は EXCLUDE_PREFIXES で完全保護されているので、Obsidian 側でいくら削除/編集しても VPS は影響ゼロ。

## 8. 実装フェーズ 4: ダミー board でガクコ承認動線の総合テスト

### 8.1 1 回目のテスト失敗(bot.py 古 SKILL 保持)

VPS の `garden/board/pending/` に `2026-06-02-dummy-test-s27.md`(`scheduled_send: 2099-12-31` で実配信ブロック)を配置。1 分以内に send_pending → notify_pending → 強化版承認依頼通知が Discord master に届く。

ガクチョが Discord で「ダミーテストの中身見せて」とガクコに伝える → ガクコは「ダミーテストが何を指しているか手元では特定できませんでした」と応答。

原因 2 つ判明:
1. **bot.py は起動時に SKILL を `open()` で読み込み固定**:更新版 daily-pilot SKILL(Mode 5 への参照あり)が反映されていない
2. **ガクコは Discord master の通知履歴を読めない**:`on_message` で `if message.author == client.user: return` なので、自分発信のメッセージ(notify_master 経由の承認依頼通知)が history に入らない

### 8.2 bot.py の修正

`list_pending_boards()` 関数を新設し、`build_dialogue_prompt` で **prompt に pending board のファイル一覧を事前注入**。これでガクコは「現在 pending にある board は何か」を知った状態で対話できる。Bash/Glob が disallowed の制約下でも、自然言語の「ダミーテスト」「シフト募集」等から該当ファイル名を絞り込める。

### 8.3 settings.json 拡張

`~/.claude/settings.json` の `permissions.allow` に **`Write/Edit(//home/vps-harappa/garden/board/**)`** を追加。新パス board への Edit 権限が無いとガクコが status を書き換えられない。

### 8.4 bot 再起動の落とし穴

`run-bot.sh` を手動で叩いた直後、cron `*/2 * * * *` の keep-alive も同時に走って **bot.py が二重起動** する事案発生。`pkill -f "python bot.py"` で全部殺してから `run-bot.sh` を 1 回だけ叩いて 1 プロセス化。

### 8.5 2 回目のテスト成功

ガクチョ報告:「すばらしい。完璧に機能しています。」

確認:
- ダミー board の frontmatter は **`status: approved`** に書き換わっていた → ガクコが Mode 5 通り正しく Edit

- → quarantine/ に手動移動して掃除完了

完全に意図通り動作。SKILL Mode 5 + list_pending_boards 注入 + settings.json 拡張の組み合わせで「Discord 自然言語 → ガクコ → board frontmatter Edit」のフルパスが成立した。

## 10. 設計判断のメモ

### 10.1 LiveSync の ignore UI を諦めて daemon フィルタへ切り替えた判断

当初は「Obsidian Self-hosted LiveSync の ignore patterns を設定する」案で進めようとしたが、ガクチョのスクリーンショットで設定 UI が 3 回切り替わっても該当項目が見つからない。LiveSync の UI はバージョン毎に大きく異なり、項目名が日本語化されている事情もあって「あと 1 タブ試して見つからなければ代替案へ」と判断 → daemon に EXCLUDE_PREFIXES を入れる形に切り替え。

結果として daemon フィルタの方が **構造的に頑健**(Obsidian UI 設定に依存せず、VPS 側のコードで完全制御 / 別端末が増えても自動的に同じルールが適用)。当初の案より上等になった。

### 10.2 board と log をセットで vault 外に出した判断

ガクチョの問い「VPS で完結するなら vault でファイルを管理する必要ある?」が起点。当初は board だけ移す案だったが、log も含めて vault 外にしたほうがセマンティクスが揃う(`garden-mirror/` = Obsidian 共有 / `garden/` = VPS スクリプト)。

工数差は 30 分程度。後で log だけ追加移行すると ADR を書き直す手間 + 参照箇所の二度更新 → 一気にやる方が安い。S25 学び「同じ問題を別角度から照らす提案は 1 ファイルに統合」の応用。

### 10.3 list_pending_boards で prompt に事前注入する判断

bot.py の Claude には Bash/Glob/Grep が disallowed。ディレクトリ ls ができない。これに対する選択肢:
- A. Bash の path-scoped allow を追加 → settings.json 拡張で対応可能だが、disallowedTools の Bash 全禁止と矛盾
- B. ガクコ専用の helper を bot.py に組み込み → これを採用
- C. ユーザがファイル名を具体的に指定 → ユーザビリティ低下

B 採用。bot.py が Python で `os.listdir` した結果を prompt に注入する形で、Claude はリストを「与えられた事実」として扱える。Bash/Glob 全禁止を維持しつつ実用性確保。

### 10.4 ダミー board の scheduled_send を 2099-12 にした判断

実配信(LINE / Discord master プレビュー)が走ると Discord に余計なノイズが出る。send_pending.py は scheduled_send が未来なら dispatch を skip するロジックなので、2099-12-31 にしておけば「承認動線だけ」をテストできる + processed/ に移動しないので承認後に board の最終状態を確認可能。

実テスト後の board は frontmatter が `status: approved` に書き換わっていて、ガクコ Mode 5 の動作確認証跡として完璧に残った。

### 10.5 docker-compose v1 の `KeyError: ContainerConfig` への対処

docker-compose v1.29 と Docker engine v23+ の既知の互換問題。`up -d --build` が失敗するが、`down` してから `build --no-cache` してから `up -d` のサイクルで回避できる。VPS のセットアップ全体を docker-compose v2 plugin に移行するのは Phase 3b 後追い宿題に残置。

## 11. 学び

- **想定 UX の検証は構造を単純化する** — 「ガクチョが Obsidian で board を承認」想定が事故の温床だった。ガクチョの「Discord でガクコに承認する方が楽」の発話が、設計を一段抜本化させた。同じパターンは S26 でも soil で起きた(「ガクチョが Obsidian で soil を書く」想定→実態は Claude 経由)。**「ガクチョはどう運用したいか」を先に聞く** ことの価値
- **代替案への切り替えタイミング** — LiveSync UI 探しに 3 回失敗した時点で「あと 1 タブで諦めて代替に切り替える」と宣言したのは正解。結果として daemon フィルタの方が頑健になった。**手詰まりを「もうちょっと試そう」と引きずらない判断** が時間の質を上げる
- **bot.py の起動時固定読み込みは罠** — `open()` で SKILL を読み込み固定すると、SKILL を更新しても再起動するまで反映されない。実装当時は意識していなかった落とし穴。新しい SKILL/prompt 仕組みを足す時は **bot.py の再起動が必要**(または file watcher で再読み込み)と覚える
- **二重起動チェックは run-bot.sh だけでは不十分** — PIDFILE の `kill -0` チェックは race condition で抜ける。`pkill -f` で先に確実に殺してから 1 回だけ起動するパターンが安全
- **daemon フィルタが LiveSync ignore より上位概念** — Obsidian 側の設定は端末毎にバラつく可能性があるが、daemon は VPS 側の単一の真実。「クライアント設定で防ぐ」より「サーバ側で構造的に遮断する」方が頑健
- **scheduled_send 遠未来でのテスト戦略** — 配信動線の確認だけして実配信を避けるテクニック。新規シナリオの動作確認に再利用可能
- **CouchDB 直接削除は最後の手段** — Obsidian UI で消えない場合の最終手段だが、LiveSync 経由でちゃんと「削除」を伝播させるためには各端末で sync が必要。**手動で削除してもらうのが結局一番速かった**
