# セッション46(2026-06-15)— sns_manager 使用済みフォルダ仕様 + 実投稿バグ根治 + 初回フロー通し実行

> S45 で本番稼働まで作った `sns_manager` を、ガクチョと**実機で初めて通し**たセッション。
> 途中で **S45 の潜在バグ(実投稿が 100% 失敗する契約不一致)を発見・根治**し、火・土 2 本を IG+FB に実予約するところまで完走した。

## きっかけ

ガクチョ「SNS アップをやろうとしている。Drive はこのフォルダで OK? inbox/proceeded のようなフォルダ分けがされていないが、ここに直接画像を置く仕様?」

→ 仕様確認すると「フォルダ直置き(サブフォルダ分けなし)」で正しいが、**使用済み画像の管理が無い**穴を発見。`fetch-images --week` の `--week` はローカル保存先名にしか使われず、Drive 側は毎週**全画像を候補に出し続ける**。ガクチョ判断「運用カバーはしない。**作成に使った画像は使用済みフォルダに自動で移動する仕様**にして」。

## やったこと

### A. 使用済みフォルダ自動退避(新仕様)

候補は「フォルダ直置き・置きっぱなし」で OK。**投稿予約に成功した画像の Drive 原本を `使用済み/` サブフォルダへ自動 move** する。差し替えで不採用になった候補は移動せず候補のまま残る(= 翌週も候補)。

- [lib/drive_reader.py](../../garden/services/sns-manager/lib/drive_reader.py) — `find_image_by_name` / `ensure_used_folder` / `move_to_used` を追加。`list_images` はフォルダ**直下のみ**走査なので `使用済み/` 内は自然に候補から除外。
- [processor.py](../../garden/services/sns-manager/processor.py) — `schedule` 成功後に `_archive_used_images()`(ローカル名で Drive 候補を引き当て move)。**予約は完了済みなので move 失敗は warning 止まり**。`--no-archive` で抑止可。
- 認証は full `drive` scope なので API 的に move/create 可能。**書き込み権限テスト GREEN**(`使用済み/` 実作成。SA は編集者共有)。
- SKILL Mode A2 / monday-caption-draft 種に挙動を明記。

### B. ★S45 潜在バグの発見・根治(実投稿の契約不一致)

初の実予約で `ig_scheduler` が **500**。切り分けると実体は **422 バリデーションエラー**(その内部に画像バイト 0xff が混ざりシリアライズ失敗 → 500 化)。

- **根因**: 稼働中の `ig_scheduler`(6週間 Up)は **JSON で「画像の公開URL(`image_url`/`image_urls`)+ caption + publish_at(tz付き)」を受け取る**契約(投稿時刻に Meta がその URL を取得して即時投稿。**IG専用**、`ig_photo`/`ig_carousel` のみ)。一方 Garden の `meta_client` は S45 で「**画像バイナリを multipart 送信**(サーバーがホスト)新方式」に**クライアントだけ**書き換えており、サーバー未対応で噛み合わず。
- **S45 のスモークは dry-run までだった**ため、実投稿経路は今回が初検証 → ここで露見。**実投稿は 100% 失敗する状態**だった。
- **修正** = [lib/meta_client.py](../../garden/services/sns-manager/lib/meta_client.py) `_post_schedule_with_images` を **URL ベース**に戻す:`upload_photo_for_url()`(画像を FB ページに非公開アップロード → Meta 取得可能な CDN URL を得る)で公開 URL を作り、`/schedule` に **JSON**(`platform`/`image_url`|`image_urls`/`caption`/`publish_at`)で送る。HMC 実績の方式と同形。
- **FB は別系で元から正常**:`schedule_fb_photo`/`schedule_fb_carousel` が Graph API へバイナリを直接 `scheduled_publish_time` 予約(ig_scheduler 非依存)。

## 学び・メモ

- **dry-run スモークは「実契約」を検証しない**。S45 の schedule は dry-run GREEN だったが、実 API では client/server 契約不一致で全失敗。**外部投稿系は最低 1 本を実機で通すまで「動く」と言えない**(= 今回月曜前倒しで通した価値)。
- **creative パイプライン自体は設計通り機能**(画像 Read → 目的割付 → 一言起点の文体生成 → ガクチョ赤入れ)。文案は2本とも全面差し替えになったが、これは「人間が著者・Garden が整形者」原則の健全な作動。

## 要観察(次の宿題)

- IG は **FB CDN URL を保存して投稿時刻に Meta が取得**する方式。CDN URL の寿命次第で、特に **土(5日先)の job 12** が投稿時に URL 失効で失敗する理論リスク。**火 6/16 20:00 / 土 6/20 08:00 に IG+FB に実際に出たか**を当日確認(失敗時 ig_scheduler が `error_msg` + メール通知)。
- これが通れば sns_manager は実投稿実績ありとして **test→active 昇格**判断材料が揃う。
