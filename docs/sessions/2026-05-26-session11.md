参加者: 塚越さん(ガクチョー)、Claude (Opus 4.7)

## 開始時の状態

- セッション10 末で Phase 3a A-2 完了(CouchDB + LiveSync 三端末同期動作)
- 次回本命候補: (a) A-3 平文 MD ミラー daemon / (b) A-1 本番ランチャー / (c) 連絡板 / (d) gakuchovault 内 Garden フォルダ設計 / (e) スキーマ拡張 5 項目 ADR 化

→ Phase 3a 本命候補は **次回送り**、本セッションは **VPS 管理体制の整備** に集中

## 流れ

1. (a)〜(e) の内容を一行ずつ整理 + ガクコのスコープ確認 + VPS 設定の git 保存状況の現状把握を提示
2. 塚越さんから VPS 管理体制を要件3つで定義:
   - ステータスがドキュメント化される
   - 別リポの AI も状況が把握できる
   - GitHub にプッシュされ、トラブル時に復旧できる
3. VPS の棚卸し(ssh 経由)実施:
   - ガクコ系: ローカル正本 + GitHub 経由配信 → 既に確立
   - その他系(NPM / ig_scheduler / proxy-manager / cron / garden services)→ git 管理不在 or 不徹底が判明
   - 特に NPM 内部 DB(Proxy Host / SSL)が手動 UI 投入のみで復旧手段なし
4. 4つの方針質問に塚越さん回答:
   - 管理範囲: **ハラッパ直結 + ig_scheduler を本 repo の `vps/` に集約**(ig_scheduler は HMC sns_pilot の一環)
   - ガクコ: **別 repo 継続 + 本 repo から参照のみ**(submodule 不採用)
   - NPM: **定期 export スクリプト + 本 repo に commit**
   - secret: **現状継続**(ローカル平文 + git 除外)
5. 進め方の確認 → 塚越さん「**3 をやってしまおう。寄り道はスピーディに終わらせたい**」(=ADR + ドキュメント骨格 + NPM export 実装 + ig_scheduler 取り込み + cron スナップショット 全部)
6. 並列で実装:
   - ADR `docs/decisions/2026-05-26-vps-management-policy.md` 起草
   - `vps/README.md` / `dev-flow.md` / `recovery.md` 起草
   - `vps/proxy-manager/` 一式(docker-compose ミラー + export.sh + README + .gitignore)
   - `vps/ig_scheduler/` 一式(VPS から scp で取り込み: docker-compose / Dockerfile / app.py / requirements.txt / .env.example + README + .gitignore)
   - `vps/cron/crontab.snapshot` 取得
7. 初回 NPM export 試行 → sudo パスワード要求でエラー
8. `export.sh` を alpine コンテナ経由(`docker run --rm alpine sh -c 'tar...'`)に書き直し → 1.3MB の backup 取得成功
9. 本 repo `.gitignore` 更新、ガクコ repo `README.md` に1行追加(本 repo `vps/` への参照)
10. `garden/MAP.md` 更新(現在地・区画・Phase 3b・宿題・決定索引・直近セッション)

## 動作確認結果

```

        サイズ: 1.3MB(data + letsencrypt、272 ファイル)
        git check-ignore で除外確認済
```

- ガクコ repo README に「VPS 全体管理は harappa-garden の vps/ を参照」を追記

## 議論したが結論未定

### NPM backup の logs 除外

- 現状 backup 272 ファイル中の多くが logs(`data/logs/*.log.*.gz`)
- 復旧に不要なため `export.sh` に `--exclude='data/logs/*'` を入れたい
- ただし「ログも復旧の参考になる場合がある」可能性 → Phase 3b で判断

### NPM backup の暗号化 → GitHub push

- 現状ローカル WSL のみで保管(WSL 全壊で消失)
- age / sops 等で暗号化して GitHub に push する経路は将来検討
- 今は secret 全般の WSL 一極集中問題と同じ信頼境界

### gaku-co-oc(openclaw)のローカル差分

- VPS 上 `~/gaku-co-oc/` に未コミット変更が複数(`M src/discord/monitor/*` 他)
- 上流 PR 化 or 内部 repo 化が望ましいが、本セッションではスコープ外
- Phase 3b で判断予定
