## 1. soil 正本ルール明文化(宿題 3 = S23 持ち越し)

### 1.1 実態調査

S23 で `soil/` が repo / vault / VPS garden-mirror の 3 箇所配置になったあと、運用ルール未整理のまま 1 週間経過していた。実態を見ると:

3 つの書き手(Claude セッション / 菌糸 VPS / ガクチョ Obsidian)× 3 箇所コピー で 9 経路の可能性、まだ運に守られている状態。

### 1.2 想定 UX と実態のズレ — ガクチョの問い

私が「ガクチョが Obsidian で soil を編集するシーン」を 6 つ列挙したところで、ガクチョから核心の問い:

> 「僕が soil を直接編集するシーンがあまり浮かばないんだよね。編集するとしても Claude Code に相談しつつになると思う」

→ **設計が一段単純化**。書き手は **2 つだけ**(Claude セッション + VPS 上のエージェント群)、vault は **読み取り専用ビュー** に再定位される。

### 1.3 採用案: 層別正本 + 単方向 rsync 2 本

- 処理完了時に `last_ingested_at` を **付与のみ**(以降そのファイルは触らない)

## 5. 設計判断のメモ

### memory_logger.py を拡張しないという判断

S23 で「memory_logger.py を `last_ingested_at` 対応に拡張」と書いていたが、実装段階で「拡張不要」と判断。理由:

- memory_logger.py = RAW append 専用、シンプルが正義
- `last_ingested_at` の付与は ingest-raw seed prompt(Claude Code 経由)の責務に閉じ込められる
- ingest-raw が初期 frontmatter に項目を追加する形のほうが、責務境界が明確

S23 の宿題リストを字義通り実装せず、設計意図を優先して構造を保った例。

### turn 単位冪等 vs RAW 単位冪等

S23 4 論点合意では「RAW 単位 `last_ingested_at` で turn 単位の冪等保証」と書いたが、dry-run で turn 時刻比較が脆いと判明。

理由を分解すると:
- frontmatter `last_ingested_at: 2026-06-02T03:30:00+09:00`(固定の時刻)
- RAW 内 turn 見出し `## 07:42:15`(HH:MM:SS のみ、日付なし)
- LLM が「ファイル日付 + turn 時刻」を組み立てて比較するロジックは LLM の柔軟性に依存
- 結果として「03:30 < 07:42 → new」と判定してしまう

設計上の選択:
- **A: turn 見出しに完全日時を書く**(memory_logger.py 変更)→ 過去 RAW の遡及修正必要
- **B: last_ingested_at を実行時刻にする**(03:30 ではなく実時刻)→ 結局当日 turn の問題が残る
- **C: RAW 単位の処理完了マーカーに変更**(turn 比較を廃止)→ 翌日反映遅延が発生

C を採用。ADR §6「夜間バッチで前日分」と整合的で、LLM の柔軟性に依存しない確実な冪等性が得られる。

### dry-run 4 回を回した価値

- 1 回目で実走を確認できたのは大きい(SKILL Mode 1 の設計が機能している証明)
- 2-3 回目で冪等性脆弱を発見できたのは、本番 cron 前に問題を捕まえる絶好の機会
- 4 回目で完全 skip を確認したのは、修正の正当性を保証
- 「動いた」で止めず、4 回回したことで設計の脆さを表面化できた

### 「同じ問題を別角度から」の S25 学びの実践

S25 学び:「同じ問題を別角度から照らす提案は 1 ファイルに統合」。

S26 でも soil 3 箇所配置の 3 つの懸念(正本所在 / 同期メカニズム / セッションプロトコル)を 1 ADR + 1 サービス + 1 セッションプロトコルに統合した。3 ファイル別 ADR にしなかったのは S25 学びの直接適用。

## 学び

- **想定 UX と実態の検証は設計を単純化する** — soil 編集で「ガクチョが Obsidian で書く」想定を私が抱えていたが、ガクチョの問いで実態が露呈 → 案 A(vault 正本)棄却 → 層別案がすっきり成立。「あなたはどんなシーンを想定していた?」という問いはレビュー武器
- **dry-run は 4 回回す価値がある** — 1 回目で動いても、2-3 回目で脆弱を発見できる確率は高い。冪等性は実走で確認しないと机上設計だけでは見えない
- **設計合意ファイルの字義通り実装に縛られない** — S23 合意「memory_logger.py 拡張」を実装段階で「不要」に再判断したのは正解。責務境界を優先
- **副次発見は本筋を止めずに記録する** — memory の同期穴を Stage A.5 dry-run 中に発見したが、即対応(scp 1 行)で進捗を止めず、構造論は S27 宿題に書いて先送りした
- **「同じ問題を別角度」の S25 学びは継続適用** — soil の 3 つの懸念を 1 ADR に統合
