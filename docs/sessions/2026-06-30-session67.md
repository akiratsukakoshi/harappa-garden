## 1. OAuth リフレッシュ 3 層防御の実装

### 背景
ガクチョの依頼:「リフレッシュトークンがきちんと機能するようにしてほしい」。VPS の `claude -p` 認証が約 8 時間で切れ、bot が応答不能になる問題。S59 で `setup-token`（1 年有効）に切替済みだが、サブスク OAuth トークン（短命）の内部リフレッシュが確実に動く保証がなかった。

### 調査
Claude CLI バイナリを解析し、内部のリフレッシュ処理を確認:
- `_Md()` → `N6r()` → `Bte(refreshToken, ...)` で 401 時にリフレッシュ試行
- `oauth_401_no_refresh_token_bg_worker` / `oauth_401_no_refresh_token_interactive` の 2 パスが存在
- OAuth エンドポイント: `client_id = https://claude.ai/oauth/claude-code-client-metadata`、`grant_types: ["authorization_code", "refresh_token"]`

### 実装した 3 層防御

| 層 | 仕組み | ファイル |
|---|---|---|
| 1. CLI 内部 | CLI 自身の 401 → refresh_token フロー（元から存在） | Claude CLI バイナリ |
| 2. runner リトライ | `_exec()` → auth エラー検出 → `_try_refresh()`（haiku ワンショット）→ 再実行 | `brain/runner.py` |
| 3. cron 予防リフレッシュ | 4 時間おきに CLI を叩き、トークンが有効なうちにリフレッシュを促す | `check-auth.sh` |

#### `brain/runner.py` の変更
- `run_spec()` から `_exec()` を分離（subprocess 実行の中立メソッド）
- `_try_refresh()` を新設（`claude -p "respond with only the word ok" --model haiku` で CLI 内部リフレッシュを誘発）
- auth エラー時にリフレッシュ → リトライの 2 段構え

#### `check-auth.sh` の新設
- `.env` を source して環境変数を取得
- Python でトークン残り時間を確認
- CLI ワンショットでリフレッシュを促す
- credentials.json の mtime 比較でリフレッシュの発生を検出
- **成功時は沈黙、失敗時のみ Discord 通知**（DISCORD_WEBHOOK_URL が必要、未設定）
- cron: `0 */4 * * *` で VPS に登録済み

### デプロイ
- `runner.py`、`check-auth.sh` を VPS に scp
- bot 再起動で runner.py 反映
- cron 初回実行確認: `token still valid (7.8h), no refresh needed` ✅

## 2. 請求書処理の手動テスト

### テスト経緯
ガクチョが Discord で「請求書を処理して」→ 以下が起きた:

1. **1 回目（23:24）**: インボックスに残っていた古い請求書が処理された（41 行 → `202606` タブ）。ダミー 2 件はまだ Gmail に届いていなかった（タイミング問題）
2. **ダミー再送**: ガクチョが件名を「請求書」を含む形に変えて再送
3. **2 回目（23:44）**: タイムアウト（300 秒超え）。Claude は処理を完了して CSV を作成したが、bot が結果を受け取れず
4. **3 回目（23:57）**: タイムアウト延長後。ダミー 2 件（和田祐司・グリーンフィールド・デザイン）を含む 8 行を正常処理 ✅

### 発見事項
- Gmail fetch のキーワードフィルタ（`FILTER_SUBJECT_KEYWORDS`）は正しく動作
- ダミーが拾われなかったのは**タイミング**（23:24 時点で Gmail にまだ索引されていなかった）
- タイムアウト 300 秒は請求書処理には不足

### 後始末
- 1 回目・3 回目の `202606` タブを削除（ガクチョが手動 + スクリプト）
- working CSV を削除（1 回目 `invoices_20260630_224836.csv` + 2 回目 `invoices_20260630_234205.csv` + 3 回目 `invoices_20260630_235511.csv`）
- board は `status: rejected` で processed/ に残置（害なし）
- VPS の `gmail_debug.py`（調査用一時スクリプト）を削除

## 3. Discord タイムアウト延長

### 問題
bot.py の `run_claude()` のタイムアウトが 300 秒（5 分）。請求書処理は Gmail fetch → Gemini PDF 解析 → soil 照合 → Sheets 書き込みで 5 分を超える。

### 対応
- `bot.py` のタイムアウトを **300 秒 → 600 秒（10 分）**に延長
- コメントを実態に合わせて更新
- VPS にデプロイ、bot 再起動

## 4. board_registry に手動 invoice 種を登録

### 問題
手動で invoice_processor を起動した際の種名 `invoice_processor (手動)` が `board_registry.py` に未登録 → board_lint が ERROR を通知。

### 対応
`board_registry.py` の REGISTRY に追加:
```python
"invoice_processor (手動)": {
    "kind": "🧾 請求ドラフト", "title": "{target_month} の請求登録(手動)",
    "approval": CONVERSATIONAL, "dispatcher": None},
```
