## 6. board 運用の一元管理(本セッションの本丸)

ガクチョ「種ごと管理だとまたルール無視が作られる。一元管理しないと」→ **「正本 + レジストリ + リンター + 入口」の4層**で構築([ADR](../decisions/2026-06-22-board-central-management.md))。

| 層 | 実体 |
|---|---|
| 正本(ルール) | [`garden/board/README.md`](../../garden/board/README.md) |
| レジストリ(データ) | [`board_registry.py`](../../garden/services/garden-gaku-co/board_registry.py) — 種→{種別,タイトル,承認モデル,配信先}。**新種=1行** |
| エンジン(実行) | `send_pending.py` — 分類・通知・片付け・配信を registry 由来に統合 |
| リンター(enforcement) | [`board_lint.py`](../../garden/services/garden-gaku-co/board_lint.py) — 違反検出→番人 Discord + ダッシュボード banner |
| 入口 | `plot_gardener` SKILL に「registry 登録 + lint 通過」を必須化 |

- **承認3モデル明文化**: AUTO(承認で send_pending 自動実行=シフト)/ CONVERSATIONAL(会話承認=SNS・経費・請求。approved にしても失敗ループせず archive)/ FYI(通知のみ=録音 digest・財務・朝 Triage)。**status:approved 罠を根治**。
- **不可侵原則**: 古いデータにフォールバックしない / pending=承認待ちだけ / 日本語タイトル必須 / 必ず registry 登録。
- 派生して片付いた board UX 課題(ガクチョ指摘):
  - 通知の「種別 unknown・タイトル種名のみ」→ **種別ラベル + 日本語タイトル**(registry/`title:` 由来)。
  - `pending/` に処理済みが居座る → **`relocate_terminal_boards()`(毎分)で processed/ へ自動片付け**。pending=承認待ちだけに。
  - 朝ブリーフィングの承認待ち一覧を status=pending で絞り新文面に。
- **リンターが即仕事**: 導入直後に registry 未登録の board生成種を4本検出(finance記帳/監査・朝ブリーフィング・シフト集計)→ 全登録。「種ごと管理だと漏れる」をその場で証明・解消。
