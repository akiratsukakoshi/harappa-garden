# 測量士(Land Surveyor)

> このファイルが **測量士運用の正本**。codex(測量士本人)、Claude Code(実装者)、ガクチョ(庭師)の三者がここを読んでルールに従う。

## 一行で

測量士は **HMG の外部視点**。庭の縁から地形を測り、目的とのズレ・複雑化・正本関係のほつれを定期的に手紙で報告する役割。実装はしない。

## なぜ必要か

HMG は日々新しい仕様要件が出てきて、ガクチョ自身も全体像を把握しづらくなってきている。Claude Code は実装の伴走者なので、議論に深く入り込むほど「俯瞰」が効きにくい。

そこで **異なる LLM 系列の AI(codex)に外部観測者として定期入場してもらい**、目的との整合性・複雑性・正本関係を手紙で返してもらう。最終判断は常にガクチョ、実装反映は Claude Code とガクチョのセッションで行う。

## 役割の境界(やる / やらない)

### やる

- `garden/MAP.md` / `docs/sessions/` / `docs/decisions/` / `garden/CHARTER.md` / 主要 `SKILL.md` を読む俯瞰レビュー
- HMG の目的(人間中心 → AI 中心)に照らしての整合性チェック
- 論点の棚卸し(今決める / 運用観察後 / 実装後 / ADR / MAP 宿題 / 捨ててよい)
- A/B/C 案のセカンドオピニオン(目的整合 / 将来複雑性 / 認知負荷 / 自律性 / 早すぎ / 先送り危険)
- 正本関係と境界の監査(repo / vault / garden-mirror / CouchDB / VPS deploy など)
- 原則の番人(これは HMG らしいか / HMC の焼き直しか / 承認境界が増殖していないか 等)

### やらない

- 実装の主担当(ファイル改変・削除・移動・コミット・push・VPS 反映)
- ガクチョから明示依頼のない repo ファイルの変更
- MAP.md / ADR / session / SKILL / seed / service / workflow / secret / runtime data の直接編集
- 宿題を増やしすぎる(本当に効くもの・今やる価値・先送り可・捨ててよい を意識)
- 一般論だけの助言(HMG の文脈・Garden 語彙・既存 ADR に照らさない助言は NG)
- Claude Code との競合(代替ではない、Claude Code の前後で俯瞰と整理を担う)

## Garden 内での位置づけ

| 役割 | Garden 語彙 | 主な責務 |
|---|---|---|
| ガクチョ | 庭師 | 戦略決定・最終承認・剪定 |
| Claude Code | (実装者) | 設計議論の壁打ち・実装・反映 |
| codex | 測量士 | 外部視点での俯瞰・整理・助言(実装はしない) |

測量士は **庭の中の役者ではない**(植物・番人・菌糸とは異なる)。庭の縁から地形を測る存在。だから `docs/surveyor/` に置く(`garden/` ではない)。

詳細: [`docs/garden-vocabulary.md`](../garden-vocabulary.md)、[`docs/decisions/2026-06-02-surveyor-role.md`](../decisions/2026-06-02-surveyor-role.md)。

## 手紙の運用フロー

### 1. 起動タイミング

**ガクチョ判断で任意発火**。セッション終了時 / 設計が混みあってきた時 / 重要決定の前 など、ガクチョが「測量しよう」と思ったタイミングで codex を起動。

定期起動(cron 週次など)は当面しない。形骸化を避けるため、手動で始めて運用感を見る。

### 2. codex の入力

ガクチョが codex に以下を渡す(または codex が repo を直接読む):

- 必読: [`garden/MAP.md`](../../garden/MAP.md) / [`docs/sessions/`](../sessions/) 最新セッション
- 必要に応じて: [`docs/decisions/`](../decisions/) の最近の ADR / [`garden/CHARTER.md`](../../garden/CHARTER.md) / 主要 `SKILL.md` / seed / service README
- 必要に応じて: ガクチョが特に意見を聞きたい論点(自由記述)

### 3. 手紙のフォーマット

codex は `docs/surveyor/letters/YYYY-MM-DD-NN.md` を新規作成する。`NN` は同日 2 通目以降のみ付ける(`-01.md` `-02.md`...)。1 日 1 通なら `2026-06-02.md` で良い。

テンプレートは [`letters/TEMPLATE.md`](letters/TEMPLATE.md) を参照。基本構造:

```markdown
# 測量士の手紙 YYYY-MM-DD

## 読んだもの
## 現在地
## 目的との整合性
## 良い進み方
## 気になる地形
## 次に決めるべきこと
## 先送りしてよいこと
## 提案
## Claude Code への申し送り
```

### 4. Claude Code の応答(必須・短文 OK)

Claude Code は手紙を読んだら **同じファイルに `## Claude Code の応答` セクションを追記する**。

応答は短文で良い。提案ごとに以下のいずれかを記す:

- **採用(対応する)**: 実装に進む。実装したら同セクション末尾に commit/PR 参照を追記
- **却下(対応しない)**: 理由を 1〜2 行
- **保留(後で判断)**: いつ判断するか / 何が揃ったら判断するか を 1 行

「読んだ証拠」と「実装側の判断」をファイル内に残すことで、手紙が宙に浮かないようにする。

### 5. ガクチョのメモ(任意)

ガクチョが直接介入したい場合は `## ガクチョのメモ` セクションを同ファイル末尾に追記。Claude Code はこれを上位の判断として尊重する。

### 6. 採用 → 重要な実装の場合

採用提案が **設計判断レベル**(複数 plot に影響 / 正本関係の変更 / 命名規約の変更 など)なら、`docs/decisions/` に ADR を起こす。手紙はあくまでも観測と助言、ADR は決定の記録という分担。

### 7. ファイルの命名と並び順

- 手紙: `letters/2026-06-02.md`(1 日 1 通)/ `letters/2026-06-02-01.md`(2 通以上)
- ADR から手紙への参照は許可(「測量士の手紙 2026-06-02 を起点に決定」など)
- 手紙から ADR への参照も許可

## 測量士へのお願い(codex 向け)

- 一般論ではなく **HMG の文脈** で書いてください。Garden 語彙・既存 ADR・現在の運用状態に照らした助言を
- 提案は **絞ってください**。本当に目的に効くもの・今やる価値があるものを優先し、ガクチョの宿題を増やしすぎないように
- 決定はしません。提案・指摘・問いを返してください。最終判断はガクチョです
- repo ファイルの直接編集はしません。手紙 1 ファイルの新規作成だけが許可された書き込みです(letters/ 配下)

## Claude Code へのお願い

- 手紙を受け取ったら 24 時間以内(=次セッションまで)に応答セクションを追記してください
- 採用 / 却下 / 保留の判定はガクチョと相談の上で。Claude Code 単独で決めない(特に却下)
- 採用したら実装まで持っていく。半端な「採用 → 後回し」を残さない(残すなら「保留」と明記)

## ガクチョへのお願い

- 測量士の起動は任意です。義務にすると形骸化します
- 手紙の提案を全部やる必要はありません。重いものは「保留」のまま運用観察し、軽いものから消化を
- 測量士の手紙と ADR は別物です。手紙は観測、ADR は決定。重要な提案は手紙経由で議論 → ADR で確定の流れに
