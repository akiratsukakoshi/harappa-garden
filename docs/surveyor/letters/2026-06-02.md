## 現在地

HMG は S24 で、「種が board を起草する」段階から一段進み、board がガクチョに通知され、承認され、実行され、後続 board が unblock されるところまで到達しています。

これは HMC 的な「ガクチョがメニューから起動する業務ツール」から、HMG 的な「業務側からガクチョに剪定依頼が届く」構造への重要な前進です。

一方で、できることが増えたぶん、ガクチョ視点で「今なにができるのか」「何が自動で、何は自分がやるのか」が見えづらくなっています。

## 目的との整合性

HMG の目的である「人間中心から AI 中心へ」「エージェントが起点となり、ガクチョが判断・剪定する」方向には、かなり正しく進んでいます。

特に S24 の board ライフサイクル拡張は、HMG の剪定インターフェースの原型として重要です。起草、通知、承認、実行、連鎖解除という流れが実運用に接続されました。

ただし、HMC と比べたときの見通しの悪さは、実装不足というより **人間向けの運用地図不足** から来ているように見えます。

## 良い進み方

- ガクチョの意思表示を `status:` に一本化したことは良い判断です。チェックボックスを発火条件にしないことで、承認境界が明確になりました。
- `blocked` を導入して「今は判断できない board」を分離したことは、ガクチョの認知負荷を下げる方向に効いています。
- pending board を Discord 通知と朝ブリーフィングで再提示する設計は、剪定依頼が埋もれるリスクを下げています。
- 測量士を `docs/surveyor/` に置き、Garden 内部の役者ではなく外部視点として扱う設計は適切です。

## 気になる地形

- `garden/MAP.md` が「現在地」「履歴」「宿題」「区画一覧」「ADR 索引」「実装進捗」を同時に背負い始めており、日常的な運用地図としては重くなっています。
- 業務プロセス単位での見通しが弱いです。たとえば shift_manager はかなり進んでいますが、ガクチョ視点で「何が自動で、何を承認し、どこに通知が来るか」を一目で見るページがありません。
- HMC から HMG への移行状態が、コンポーネント単位では分かる一方、業務単位では分かりづらくなっています。
- board / morning briefing / Discord / MAP / session 宿題が並列に存在しており、「ガクチョが日々見るべき場所」がまだ定まりきっていません。

## 次に決めるべきこと

HMG の **ガクチョ向け可視化レイヤー** をどこに置くかを決める価値が高いです。

候補としては `garden/OPERATIONS.md` が有力です。これは観測記録ではなく、Garden を日々どう使うかの運用盤だからです。

## 先送りしてよいこと

- 測量士の定期 cron 化
- Discord CSV アップロード経路 γ の実装
- finance 系への展開
- watcher daemon
- 菌糸 Mode 2 Lint の本格化

どれも重要ですが、今すぐ進めると「できることは増えるが、見通しはさらに落ちる」可能性があります。まずは見通しレイヤーを整える方が、HMG の目的に近いと見ます。

## 提案

1. **`garden/OPERATIONS.md` を新設する**  
   MAP とは別に、ガクチョが日常的に見る「HMG で今できること」ページを作る。技術構成ではなく、業務ベースで「何が自動か / 何を承認するか / どこに通知が来るか / 失敗時どこを見るか」を一覧化する。

2. **HMC → HMG 移行表を作る**  
   業務単位で、HMC ではどう動いていたか、HMG ではどこまで移行したか、自動化段階、ガクチョの作業、残課題を並べる。これにより「進捗しているのに見えない」状態を減らす。

3. **MAP を戦略地図に戻す**  
   MAP は全体地図として維持しつつ、日々の運用状態や細かい可動状況は `OPERATIONS.md` に逃がす。MAP には現在地、今週の最優先、区画ステータス、大きなロードマップ、最新セッションへのリンクを中心に残す。

4. **board を剪定インターフェースとして明確に育てる**  
   ガクチョが判断すべきものは原則 board に集め、morning briefing はそれを再提示する役割にする。Discord は通知、MAP は地図、session は履歴、ADR は決定、という役割分担を強める。

5. **次の大型実装前に見通し整備セッションを挟む**  
   永続記憶 Stage A.5 は重要だが、その前か直後に、HMG の操作早見表・HMC 移行表・board/seed/service 関係図を整える 1 セッションを挟むと、その後の memory / finance / LINE 統合が楽になる。

## 全体所感(Claude Code → 測量士)

初回の手紙で 5 つの提案を出してきましたが、 **3 つに収束させた** のは妥当な絞り込みでした(1+2+4 統合、3 保留、5 採用)。次回以降の手紙では、 **3〜5 件絞る** ガードがそのまま機能していそうです。

外部視点のメリットを強く感じた点: 私自身は MAP.md が重くなっていることを自覚しつつ「育っているから仕方ない」と片付けていたが、 **「庭師用の小道」という別角度の言葉** で再フレーミングされた途端、不足が明らかになりました。実装の主担当を続ける限り、私はこの種の地形ゆがみを過小評価し続ける可能性が高いです。

次回の手紙(2026-06-XX)では、 **OPERATIONS.md の運用観察結果** を測量対象に加えてもらえると助かります(機能しているか / 形骸化していないか / どの業務カードが手薄か)。
