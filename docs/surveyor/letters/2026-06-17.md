## 現在地

HMG は「区画を増やす」段階から、「区画どうしが同じ soil を読んで相互に増幅する」段階に入りました。

特に S47 finance と S48 client_steward は、単独業務の自動化ではなく、売上・打合せ・メール・請求・担当者・受注因果をつなぐ経営 OS の骨格になりつつあります。

## 目的との整合性

方向はかなり HMG らしいです。HMC の「人間が操縦席で道具を起動する」形から、Garden が soil を育て、必要な時にガクチョへ剪定依頼を出す形へ進んでいます。

特に client_steward の「生取り込みは自動、解釈は board」という境界は良いです。soil を充実させるほど Garden が賢くなる一方、確度変更・案件確定・freee 反映のような経営判断は庭師の手に残る。ここは大事な分水嶺です。

## 良い進み方

- **finance を 1 区画に統合したこと**: importer / auditor / analyzer をばらばらの技能でなく、月次財務サイクルとして束ねたのは正しい進化です。
- **Freee client の正本化**: `garden/lib/freee_client.py` + `sync-freee-client.sh --check` は、重複 API client の増殖を止める実務的な防波堤になっています。
- **Plaud のプロジェクト MCP 化**: claude.ai コネクタ依存を repo 側 `.mcp.json` に寄せた判断は、ベンダー中立方針と合っています。
- **client soil の 2 層構造**: 企業 → 案件 → meetings/emails/finance/見積/請求、という管は、後から finance・briefing・営業判断が読める良い正本です。
- **本文を外部に残す作法**: Gmail/Plaud の全文を soil に持ち込まず、要点 + 参照 ID にするのは機密・検索性・運用負荷のバランスが良いです。

## 気になる地形

### P1: finance importer は部分失敗後の再実行で二重記帳し得る

`garden/services/finance/importer.py` は行ごとに `post_manual_journal` を実行し、失敗が混じると成功済み行を Freee に残したまま `return len(failed)` します。

- 該当箇所: `garden/services/finance/importer.py` 317-368
- 成功後の input 退避は「全件成功」の後だけです: 370-374
- 失敗時メッセージは同じ review CSV の確認・再登録を促します: 362-366

このため、たとえば 79 行中 78 行成功・1 行失敗の後に同じ CSV を再実行すると、成功済み 78 行も再投稿される可能性があります。お金の書き込み系としては最優先で塞ぐべきです。

対策案は、登録前に `registration_run_id` と row hash を作り、Freee 摘要に一意マーカーを入れるか、service 側に `registered_ledger.json/csv` を残して再実行時に skip する形です。最低限、失敗時の通知は「同じ CSV をそのまま再実行しない」に変える必要があります。

### P1: auditor の未登録明細検出は 1 口座 100 件で打ち切られる

`garden/services/finance/auditor.py` の `_scan_unregistered()` は各 walletable について `limit=100` の 1 回だけ取得しています。

- 該当箇所: `garden/services/finance/auditor.py` 162-167
- Freee client 側も `get_wallet_txns()` は 1 ページ分を返すだけです: `garden/lib/freee_client.py` 386-399

月次の口座/カード明細が 100 件を超えると、status==1 の未登録明細を過少検出します。analyzer 前の地ならし役としては、ここが静かに欠けると PL の信頼性に直撃します。

対策案は `get_all_wallet_txns()` を Freee client 正本に追加し、`offset += limit` でページングすること。status 内訳も全ページ合算にしてください。

### P1: Freee refresh token の共有は reload だけでは同時 refresh 競合を防げない

S39 の改善で、refresh 前に token ファイルを再読込するようになっています。これは良いです。ただし複数 service がほぼ同時に古い refresh token を読み、同時に refresh へ進む競合はまだあり得ます。

- 該当箇所: `garden/lib/freee_client.py` 95-123

Freee の refresh token がローテートされる前提なら、共有 token ファイルには file lock が必要です。`fcntl.flock` で refresh + save の区間を排他し、lock 取得後にもう一度 disk token を読むのが安全です。

### P2: client_steward の watermark は取りこぼしを起こし得る

`sweep_client.py` は Gmail threads を `maxResults=40` で取得し、`--commit-watermark` 時に現在時刻を保存します。

- 取得上限: `garden/services/client-steward/sweep_client.py` 132-137
- watermark 保存: 245-249
- 取得結果に関わらず保存: 281-282

週次 sweep で 40 スレッドを超えた場合、41 件目以降を処理しないまま watermark が進み、次回以降に落ちます。API 途中失敗や一部 client だけ失敗した場合も同様です。

対策案は、Gmail の `nextPageToken` を最後まで辿ること、処理した message/thread の最大 `last_dt` を client ごとに保存すること、上限到達時は watermark を進めず warning にすることです。

### P2: client_steward のシグナル抽出が件名だけに寄りすぎている

finance / schedule signal は `Subject` のみを見ています。

- 該当箇所: `garden/services/client-steward/sweep_client.py` 168-170

実際の「請求書添付しました」「7/23 でお願いします」「支払処理します」は本文側に出ることが多いので、今のままだと大事な変化を見落とします。MVP としては成立していますが、test 昇格時には `format=full` ではなくても snippet 取得、または最新 1-2 通だけ body plain text を短く読む程度の拡張が効きます。

### P2: Python service の重要ロジックにテストが薄い

確認できた自動テストは主に `garden/services/launcher/test/launcher.test.mjs` でした。finance importer/auditor、client_steward は実データ dry-run では検証されていますが、再実行・ページング・watermark のような事故パターンを固定する小さな単体テストがありません。

今回のように Garden が業務の正本へ近づくほど、API 実走より前の「壊してはいけない挙動」を少数のテストで縛る価値が上がっています。

## soil / client_steward について

ガクチョの感触どおり、client soil はかなり複利的に効く資産です。理由は「情報を増やす」ではなく、「次の行動が安くなる」からです。

MTI で起きたことは象徴的です。

- Plaud は「何を話したか」を持つ
- Gmail は「誰がいつ何を確定したか」を持つ
- finance は「お金がどう実体化したか」を持つ
- soil はそれらを「案件の時間軸」に戻す

この形になると、打合せ前ブリーフィング、未返信検知、請求/入金突合、次回提案、受注因果の学習が同じ台帳から立ち上がります。

ただし、soil を増やすほど価値が出る一方で、自由記述だけで増やすと後から機械が使いにくくなります。ここからは **「文章として読みやすい台帳」+「機械が拾える小さな事実粒」** の二層にすると伸びます。

### 持ち方の提案

1. **client event ledger を作る**

   各案件に `events.md` または `events.jsonl` 相当を置き、メール・打合せ・見積・請求・入金・成果物を同じ時系列に並べる。

   例:

   - `date`
   - `type`: meeting / email / estimate / invoice / payment / deliverable / decision / followup
   - `source`: gmail_thread_id / plaud_file_id / freee_deal_id / file path
   - `summary`
   - `confidence`
   - `needs_pruning`

   README は人間向けの編集済み台帳、event ledger は Garden 向けの増分記録、という分担です。

2. **未確定情報を first-class にする**

   「7/2 or 7/3」「大木敦の役割未確認」「freee未反映かもしれない」のような情報は、曖昧なままでも価値があります。`open_questions.md` や frontmatter の `uncertainties` に集めると、client_steward が次回 sweep / briefing で自然に回収できます。

3. **obligation ledger を持つ**

   toB 案件は「約束」「宿題」「次にこちらが返すこと」が価値の中心です。メールの要返信とは別に、案件ごとに:

   - `owner`: harappa / client
   - `due`
   - `promise`
   - `source`
   - `status`

   を持つと、Garden が「忘れない営業・忘れない納品」になります。これはタスク管理ではなく、クライアント関係の債務台帳です。

4. **finance linkage を案件の必須欄にする**

   `estimate_amount` / `invoice_amount` / `freee_deal_id` / `payment_status` / `recognized_month` / `department` を案件 README に持たせると、finance Mode A が client soil を読んで「まだ PL に入っていない売上」「請求したが入金未確認」「案件見込みと Freee 実績の差」を出せます。

5. **担当者を person ではなく relationship として見る**

   `soil/people/clients/` は人の名簿として良いですが、client_steward が本当に使いたいのは「この人はこの案件で何の役割か」です。人ページに加え、案件 README 側に `roles` を持つとよいです。

   例: `sponsor` / `coordinator` / `finance_contact` / `decision_maker` / `day_of_contact`

## 次に決めるべきこと

1. **finance 書き込み系の冪等性ルール**

   Freee に書く service は、途中失敗後の再実行で何を skip するかを共通ルールにした方が良いです。finance importer だけでなく、invoice / expense / auditor apply にも効きます。

2. **client soil の最小 schema**

   文章台帳は今のまま育てつつ、機械が読むための `events` / `obligations` / `finance_links` の最小項目だけ決めると、今後の横展開が楽になります。

3. **client_steward の test 昇格条件**

   VPS デプロイだけでなく、pagination / watermark / snippet signal / board 起草の安全策が入ったら test、soil 自動 append はさらに後、でよいと思います。

## 先送りしてよいこと

- Plaud cron 取り込みの完全自動化。MCP ブリッジは有用ですが、Gmail sweep + manual Plaud pull でもしばらく価値検証できます。
- 全クライアントの一括 bootstrap。MTI 型をもう 1-2 社で磨いてから横展開した方が、schema の手戻りが少ないです。
- client_steward を core_team に開放すること。クライアント機密と財務が混ざるので、当面 master 一本で十分です。
- 見積/請求の様式統一。現時点では「金額・発行日・宛先・freee 連携」が取れればよく、帳票デザインの統一は後でよいです。

## 提案

1. **Freee 書き込みの冪等性を先に入れる**  
   finance importer の部分失敗再実行リスクを最優先で塞ぐ。あわせて invoice / expense / auditor apply にも「run_id + row hash + registered ledger」の共通作法を置く。

2. **wallet_txns と Gmail sweep は必ずページングする**  
   どちらも「静かに取りこぼす」タイプの不具合です。件数が増えた時に壊れるので、active 昇格前に直す価値が高いです。

3. **client event ledger / obligation ledger を MTI にだけ試験導入する**  
   いきなり全社 schema 化せず、MTI の `経営研修2026` で events と obligations を 1 セット作る。使いやすければ `soil/clients/README.md` に型として昇格。

4. **client_steward の first output は board でなく digest + “提案候補” にする**  
   soil 自動 append は急がない。まずは weekly digest の末尾に「board 化候補」を出し、ガクチョ/Claude が何を board にするかを数回観察するのがよいです。

5. **重要 service だけ薄いテストを足す**  
   全面テスト化ではなく、finance importer の部分失敗、auditor のページング、client_steward の watermark の 3 本だけで十分効果があります。

## Claude Code への申し送り

次セッションでは、client_steward の横展開より先に「取りこぼさない・二重登録しない」を固めるのがよいです。soil は伸ばすほど効きますが、finance と client の接続部は静かな欠落が後で高くつきます。

### 提案 1: Freee 書き込みの冪等性を先に入れる

- 状態: **採用**(S49 実装)
- finance importer は既知の穴の「部分失敗ケース」を正確に突いていた。[importer.py:372](../../../garden/services/finance/importer.py#L372) の archive が `not failed` 条件で全件成功時のみ input 退避 → 部分失敗で input が残り、同じ CSV 再投入で成功済み行を二重記帳し得た。

### 提案 2: wallet_txns と Gmail sweep は必ずページングする

- 状態: **採用(wallet_txns)/ 保留(Gmail sweep)**

### 提案 4: client_steward の first output は board でなく digest + “提案候補” にする

- 状態: **採用(現状追認)**

### 提案 5: 重要 service だけ薄いテストを足す
