## 現在地

S57 の修正で、**board の住所問題はかなり良くなっています**。`garden/board/README.md` は「board の実体は VPS 一箇所」と明記し、scribe は local `board/pending` ではなく `scribe/outbox` を経由し、push 成功後に空にする構造へ変わりました。これは「local を見て runtime を語る」事故への正しい処方です。

一方で、Garden の中核ランタイムはまだ **Claude CLI / Claude settings に強く結合**しています。業務ロジックの多くは Python・Markdown・MCP・Google/Freee API に寄っており、移行可能な資産になっています。しかし「誰が考え、誰がファイルを書き、どの権限で tool を使うか」は Claude 前提が残っています。

結論から言うと、**今すぐ “Claude を排除” するより先に、LLM ランタイム境界を 1 枚作るべき**です。目標は「Claude から Codex への引っ越し」ではなく、「Claude / Codex / Gemini を seed 単位・channel 単位で切り替えられる Garden Runtime」です。

## 目的との整合性

HMG の目的は「人間中心の操縦席」から「エージェント中心の庭」へ移ることです。その意味で、最近の開発は HMG らしい方向へ進んでいます。

- board 一元化は、剪定境界を読みやすくしました。
- `brain/provider.py` は、対話層のベンダー依存をアダプタに閉じ込める良い土台です。
- `.mcp.json` に Plaud を置いた判断は、Claude.ai コネクタ依存を弱めています。
- finance の帳票生成は `issue_document.py` + Excel template + Markdown 正本で、LLM 非依存の良い例です。

ただし現在は、設計思想と実装面が少しズレています。`garden/seeds/README.md` には `engine: claude-code / codex / gemini-cli(将来切替)` とありますが、`launcher.mjs` は `engine` を読まず常に `CLAUDE_BIN` で `claude -p` を起動します。ここが最大の移行阻害点です。

## 良い進み方

### board の VPS 単一住所化は正しい

S57 の反省は、memory で補うのではなく構造を直した点が良いです。`garden/board/README.md` は `pending` の実体を VPS に一本化し、確認手段をダッシュボード / 朝ブリーフィング / SSH に限定しています。これは別 LLM が読んでも誤読しにくい。

### 対話層には既に中立アダプタの芽がある

`garden/services/garden-gaku-co/brain/provider.py` は `Provider.chat()`、`ToolSpec`、`ToolCall`、`LLMResponse` を定義し、`gate.py` / `respond.py` は `import anthropic` せず `Provider` だけを呼んでいます。これは Codex / OpenAI / Gemini Provider を足す余地があります。

### 業務サービスは意外と LLM 非依存になっている

Freee、Google Sheets、Drive、SNS、帳票生成などは、実行ロジックが Python に落ちています。ここは Claude 依存というより「bot や seed がどのコマンドを叩くか」の問題です。移行の主戦場は業務コード全面ではなく、ランチャー・bot・権限・ドキュメントです。

## 気になる地形

### P1: seed ランチャーが `engine` を無視して Claude CLI に固定されている

`garden/seeds/README.md` は engine の将来切替を掲げていますが、実装では `launcher.mjs` が `CLAUDE_BIN` を決め、`claude -p` 専用の引数を組み立てています。

- `garden/services/launcher/launcher.mjs` 36: `CLAUDE_BIN`
- `garden/services/launcher/launcher.mjs` 360-380: `--mcp-config` / `--permission-mode` / `--allowedTools`
- `garden/services/launcher/launcher.mjs` 456-460: MCP 引数も Claude CLI 仕様
- `garden/services/launcher/launcher.mjs` 497 付近: `spawnSync(CLAUDE_BIN, ['-p', prompt, ...])`
- `garden/seeds/*/*.md`: すべて `engine: claude-code`

この状態では、メインモデルを Codex に替えるには全 seed を一気に変える必要があり、切替ではなく引っ越しになります。まず `engine` を本当に読む `runner` 抽象が必要です。

### P1: master Discord bot が中立 Provider を使わず Claude subprocess 直結

LINE/core_team 用には `brain/provider.py` がありますが、Garden の主会話面である `bot.py` は別系統です。

- `garden/services/garden-gaku-co/bot.py` 8-13: ホストの `claude` subprocess 前提
- `garden/services/garden-gaku-co/bot.py` 33: `CLAUDE_BIN`
- `garden/services/garden-gaku-co/bot.py` 626-645: `run_claude()` が `claude -p --model sonnet` を直接実行

つまり、いちばん重要な master channel が、最も強く Claude に固定されています。Codex 移行を考えるなら、ここを `Provider` または `AgentRunner` の裏に入れるのが第一候補です。

### P1: 権限モデルが `.claude/settings.json` に隠れている

S57 の SNS temp 権限不足は、まさにここから出ています。コードや docs では「この Python entrypoint だけ許可」と書かれていますが、実際の許可正本は VPS の `/home/vps-harappa/.claude/settings.json` です。repo 外なので、別 LLM はその権限契約を読めません。

Codex へ移す時に危ないのは、Claude 固有の `allowedTools` そのものより、**権限の意図が repo 内の機械可読データになっていないこと**です。`capabilities.py` / board registry に近い形で、seed / scope / command allowlist を repo 内に置き、Claude settings はそこから生成される側にした方がよいです。

### P1: scribe は local/VPS 境界としては改善済みだが、モデル中立ではない

scribe は Plaud OAuth token の所有ホストを local WSL に固定した判断自体は妥当です。refresh token を VPS と共有しないのも安全です。

ただし現実の流れは `run-local.sh` が local で `claude -p` を起動し、Plaud MCP を Claude CLI 経由で使う形です。

- `garden/services/scribe/run-local.sh` 4-8: Plaud token は local WSL
- `garden/services/scribe/run-local.sh` 62-75: local `claude` / `node` を探して launcher 起動
- `garden/seeds/scribe/daily-recording-sweep.md` 29-43: `engine: claude-code` + Claude CLI の MCP 設定

Codex に切り替えるには、Plaud MCP を Codex 側から同じように使えるか、または local bridge が Plaud から構造化入力を取り出して、LLM には標準入力として渡す形にする必要があります。ここは全体移行の最難所です。

### P2: `bot.py` が太りすぎている

`bot.py` は会話、memory、skill load、topic routing、権限説明、各サービスの実行手順、scribe marker、Discord 送信を抱えています。Claude の調子が悪いと感じる原因の一部は、モデル性能だけでなく **prompt に詰め込む役割が増えすぎている**ことだと思います。

特に skill ごとの Bash 実行手段が長文で直書きされており、実際の command allowlist と自然言語説明がズレる余地があります。これは `tools/registry.py` / `capabilities.py` に寄せると、LLM を替えても同じ行動層を使えます。

### P2: `freee_client.py` のコピー方式は管理されているが、まだ重複地形

`garden/lib/freee_client.py` と各 service の `lib/freee_client.py` は現時点で hash 一致していました。`sync-freee-client.sh --check` もあるので「野良重複」ではありません。

ただし、今後 Codex / Claude が交互に触るなら、コピー方式は修正漏れを呼びやすいです。最低限、pre-deploy / CI / watcher で `sync-freee-client.sh --check` を必ず通すのがよいです。可能なら service rsync 時に自動同期する方が安全です。

### P2: ドキュメントの Claude 前提・古い前提が残っている

`CLAUDE.md` は名前としては入口ファイルなので残してよいですが、内容には「Claude が終了時に更新」「Claude Code が伴走」「各 SKILL は `.agent/skills`」など、現状とズレた表現が残っています。

`garden/seeds/README.md` も「すべての cron 系の種は VPS」「WSL cron は使わない」と書いていますが、scribe は例外として local WSL poll に戻りました。例外は妥当ですが、正本に反映されていないと次の LLM がまた誤ります。

## Claude / Codex 移行リスク

### すぐ壊れそうなもの

1. **cron seed 全般**  
   launcher が Claude CLI 専用なので、Codex へ切り替える入口がありません。

2. **Discord master bot**  
   `run_claude()` 直結。Provider 化されていないため、Codex 移行の影響が大きいです。

3. **Claude settings 依存の Write/Edit/Bash 許可**  
   権限の意図が repo 外 runtime にあり、移行時に再現漏れが起きます。

4. **scribe / Plaud**  
   MCP 設定は repo にありますが、実行は local Claude CLI + Plaud token という特殊構成です。

### 比較的安全なもの

1. **Markdown 正本群**  
   `garden/plots/*/SKILL.md`、soil、workflow、ADR、sessions はほぼ中立資産です。

2. **Python 業務サービス**  
   finance / expense / invoice / SNS / field / shift は、LLM から叩く wrapper を替えれば残せます。

3. **board 正本 + registry + lint**  
   ここはモデル非依存の良い基盤です。

4. **LLM 非依存の帳票生成**  
   `issue_document.py` は、今後の「LLM が考える、Python が確定処理する」型の見本です。

## 次に決めるべきこと

1. **Garden Runtime の単位を決める**

   `seed` と `master bot` を同じ抽象で見るか、別抽象にするか。私の提案は:

   - `SeedRunner`: cron / one-shot seed 用
   - `ChatProvider`: 会話応答用
   - `ToolRegistry / Capability`: 行動許可用

   の 3 層です。

2. **Claude settings を正本にするのをやめる**

   repo 内に `garden/runtime/capabilities.yml` のようなものを置き、scope / seed / command / path / host を宣言する。Claude settings は生成物にする。

3. **Codex 移行の MVP をどこに置くか**

   いきなり全 seed ではなく、次のどちらかがよいです。

   - 低リスク: `mycelium/index-refresh` など read/write が限定された seed を Codex Runner で試す
   - 効果大: Discord master bot の「会話だけ」Provider 化し、ファイル編集はまだ Claude CLI に残す

## 先送りしてよいこと

- Claude CLI の即時排除。今やると scribe / seed / bot / permissions が同時に崩れます。
- すべての SKILL 名・Garden 語彙の一括改名。まず runtime 境界を作ってからで十分です。
- Plaud の consumer API 化。MCP + local token は複雑ですが、今は動いています。先に runner 抽象で囲う方が効きます。

## 提案

### 1. `launcher.mjs` に `engine` 実装を入れる

最初の改修はここです。

- `engine: claude-code` は現状通り
- `engine: codex` を追加できる形にする
- `buildClaudeArgs()` と `buildCodexArgs()` を分ける
- `execute.mcp` は `execute.tools` / `execute.permissions` に一段抽象化する
- dry-run で「どの runner が何を起動するか」を log に出す

まだ Codex runner が未完成でも、`engine` を読んで未対応なら明示エラーにするだけで前進です。いまは schema が嘘をついています。

### 2. master bot の `run_claude()` を `AgentRunner` 化する

`bot.py` の `run_claude()` を直接消すのではなく:

- `runner = Runner.from_env()`
- `runner.run_chat(prompt, system, tools, scope="master")`

のように薄く包む。初期実装は ClaudeSubprocessRunner でよいです。その後 CodexRunner / OpenAIProvider を足せます。

### 3. repo 内に権限正本を置く

S57 の temp 権限不足を再発させないため、`settings.json` を手作業で直す運用から離れる。

候補:

- `garden/runtime/capabilities.yml`
- `garden/services/garden-gaku-co/capabilities.py` を seed / command まで拡張
- `board_registry.py` と同じ発想で、種→許可 command / writable path / host を 1 テーブル化

生成先は Claude settings でも Codex config でもよいですが、正本は repo に置くのが大事です。

### 4. scribe は「local bridge」と「LLM 判断」を分ける

Plaud token が local 固定なのはよい。ただし local bridge が Claude CLI そのものを起動する形だと、モデル切替の最大障害になります。

理想は:

1. local bridge が Plaud MCP から note/transcript metadata を取る
2. JSON/Markdown の中間入力を作る
3. 選択された LLM runner が主体判定・soil 化案を作る
4. Python または runner が outbox / soil に反映

一気にやらず、まず `run-local.sh` の責務名を「Plaud token host bridge」として明確にし、Claude runner はその内側の一実装に落とすのがよいです。

### 5. ドキュメントの役割分担を中立名に寄せる

`CLAUDE.md` は互換のため残してよいですが、内容は次のように更新候補です。

- 「Claude が」→「実装エージェントが」
- 「Claude Code ヘッドレス」→「Garden Runtime / seed runner」
- 「codex = 測量士だけ」→ 将来は「codex は測量士にも実装者にもなり得る。ただし同一セッションで役割を明示」
- `garden/seeds/README.md` に `execution_host: local` の例外と scribe を反映

これは移行時に人間と LLM の認知負荷を下げます。

## Claude Code への申し送り

今の HMG は「Claude でしか読めない知識」に依存しているというより、**Claude でしか起動できないランタイム**に依存しています。知識・業務ロジック・正本関係はかなり移行可能です。

次に手を入れるなら、機能追加ではなく以下の順がよいです。

1. `launcher.mjs` の `engine` を本当に読む
2. `bot.py` の `run_claude()` を runner 抽象へ退避
3. `.claude/settings.json` 相当の権限意図を repo 内正本にする
4. `garden/seeds/README.md` / `CLAUDE.md` / `MAP.md` の Claude 前提を「Garden Runtime」へ更新

この4つが入ると、Claude から Codex への移行ではなく、LLM を差し替えられる庭になります。

## ガクチョへの短い見立て

Opus への信頼性低下は、モデルの調子だけでなく、**Claude に任せる文脈・権限・役割が増えすぎたこと**にも原因がありそうです。HMG はかなり育っていますが、実装者 Claude が「設計者・実行者・権限管理者・runtime・記憶の読者」を兼ねすぎています。

Codex へ主担当を移すのは可能です。ただし安全な順番は「主担当を替える」より先に「主担当を替えられる構造にする」です。最初の合言葉は、Claude 排除ではなく **Garden Runtime 抽象化** がよいと思います。
