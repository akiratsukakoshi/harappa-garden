## 追記: ガクチョの目的整理を受けて

ガクチョの整理を受けて、今回の目的をもう一段正確に置き直す。

今回やりたいことは「Claude から Codex へ移行すること」そのものではない。より本質的には、Garden が特定ベンダーや特定エージェント実装に強く結びつきすぎず、必要になったときに Claude / Codex / OpenAI API / その他の runner へ、低負荷で切り替えられる状態を作ることだと思う。

つまりゴールは、単発の引っ越しではなく「移管可能性の制度化」。

- 変更すべき箇所が集約されている
- そこを確認・修正すれば全体の切替が見通せる
- 依存の残り香を検出できる
- dry-run / smoke test / rollback が手順化されている
- エージェントの役割分担ドキュメントも、ベンダー名ではなく Garden 内の役割で保守できる

S60 はこの方向の入口としてかなり良い。ただし現状はまだ「Claude を外せる構造に向かい始めた」段階であって、「いつでもベンダーを切り替えられる型」までは届いていない。次に作るべきものは、Codex runner そのものより先に Vendor Switch Kit だと思う。

## 提案: Vendor Switch Kit を Garden Runtime に置く

置き場所は `garden/runtime/` がよい。ここを「LLM / agent runner / API provider の切替正本」にする。

### 1. 正本ドキュメント `garden/runtime/VENDOR_SWITCH.md`

ここに「ベンダー切替時に必ず見る場所」を集約する。

含めたい項目は以下。

- 現在の active engine 一覧
- seed runner の切替点
- master bot runner の切替点
- API provider の切替点
- MCP / skill / memory / grants の確認点
- local / VPS の配置差分
- secret / token / subscription の確認点
- smoke test 手順
- rollback 手順
- 切替後に更新すべきドキュメント一覧

このドキュメントがあると、「Claude から Codex へ」だけでなく、「Codex から別 runner へ」も同じ型で扱える。

### 2. engine catalog `garden/runtime/engines.yml`

```yaml
active:
  seed_runner: claude-code
  master_runner: claude-code
  line_provider: anthropic

engines:
  claude-code:
    type: agent-cli
    runner: claude
    grants_renderer: render-claude-settings.py
    mcp: supported

  codex:
    type: agent-cli
    runner: codex
    grants_renderer: render-codex-settings.py
    mcp: verify

  openai-api:
    type: api-provider
    provider: openai
    tools: function_calling
```

これは実装設定そのものというより、Garden が把握している engine 台帳。現状の `seeds.yml` / `agents.yml` / `grants.yml` / `.env` を横断して「いま何が Claude 固有なのか」を見えるようにするための地図になる。

### 3. 切替チェックリスト `garden/runtime/checklists/switch-to-{engine}.md`

engine ごとに最低限の手順を固定化する。

例:

- precheck: runner binary / API key / model / subscription
- dry-run: 書き込みなしで起動できるか
- read-only run: repo 読み取りだけで期待通り動くか
- scratch write: 一時領域だけに書けるか
- seed smoke: 低リスク seed を 1 本だけ回す
- master bot smoke: 一時的に bot runner を切り替えて応答確認
- rollback: active engine を戻す、生成物を破棄する

ここまであれば「誰がやっても切替の抜け漏れが減る」。これはコードの抽象化と同じくらい価値がある。

### 4. vendor lock 監査スクリプト `garden/runtime/audit-vendor-lock.py`

このスクリプトは、切替前後に必ず走らせる。

検出対象の例:

- `claude`
- `CLAUDE_BIN`
- `.claude/settings.json`
- `--allowedTools`
- `--disallowedTools`
- `--permission-mode`
- `sonnet` / `haiku` / `opus`
- `anthropic`
- `engine: claude-code`

ただし単純に「見つかったら悪」ではない。分類が大事。

- 意図された依存: Claude runner 実装、Claude 用 grants renderer、履歴ドキュメント
- 集約済み依存: `garden/runtime/` に閉じているもの
- 未集約の依存: bot / seed / script / docs に散っていて切替時に漏れそうなもの

この分類ができると、Claude 固有語が残っていても安心できる場所と、潰すべき場所を分けられる。

### 5. smoke tests `garden/runtime/smoke/`

runner ごとに最低限の互換性テストを置く。

- runner が起動する
- prompt が渡る
- cwd が期待通り
- MCP 設定が読める、または非対応として明示される
- tool allow / deny が期待通り働く
- scratch 領域だけに書ける
- exit code / stderr が runner 層で捕捉される

Codex へ移る前にこの smoke test があると、移行判断が「なんとなく動いた」ではなく「Garden Runtime として受け入れ可能」に変わる。

## ドキュメント上の役割名もベンダー名から外す

`CLAUDE.md` や `MAP.md` の課題は、Claude という名前が「実装 runner」と「庭の内側の実装者ロール」の両方を背負っていること。

今後は役割名を以下のように分けるのがよい。

- 実装者 / Gardener: Garden 内で作業する主担当ロール
- 測量士 / Surveyor: 外部視点でレビューするロール
- Runner / Engine: Claude Code, Codex, OpenAI API などの実行基盤

こうしておけば、「実装者は Claude」ではなく「実装者ロールを現在は Claude Code runner が担っている」と表現できる。これが移管可能性の中心になる。

## 優先順位

1. `garden/runtime/VENDOR_SWITCH.md` を作る
2. `garden/runtime/engines.yml` を作る
3. `audit-vendor-lock.py` を作る
4. `CLAUDE.md` / `MAP.md` / `AGENTS.md` の役割語を runner から分離する
5. Codex runner 実装に入る
6. OpenAI API / その他 runner も同じ型で登録できるか確認する

この順番なら、Codex 移行は「目的」ではなく「移管システムの最初の検証ケース」になる。

## 測量士としての結論

今の Garden は、S60 によって vendor portability の入口に立てている。ただし、まだ「切替可能なシステム」ではなく「切替可能にするための部品が生まれた」段階。

次の一手は Codex を急いで本番投入することではなく、Vendor Switch Kit を作り、Claude から Codex への移行をその kit の初回リハーサルにすること。

合言葉にするとこう。

> ベンダー切替は実装作業ではなく、Garden Runtime の定期点検作業にする。
