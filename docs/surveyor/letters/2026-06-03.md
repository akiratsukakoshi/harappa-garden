## 現在地

HMG は基盤整備の山を越え、次のフェーズとして「社内 LINE でガクコが活動する」「業務単位で SKILL + 種を増やす」段階に入ろうとしています。

この段階で必要なのは、個別 tool を場当たり的に増やすことではなく、業務フローを HMG の plot / SKILL / 種 / service / capability に変換するための型です。

## 目的との整合性

HMG の目的は「人間がツールを選んで起動する」HMC から、「業務側が起点となり、Garden が必要な手段を選び、ガクチョが判断・剪定する」構造へ移ることです。

その意味で、ガクチョが今回言語化した「どんなツールを用意し使うかは Garden 側に任せたい。こちらは workflow に当てはめて起こしてほしい SKILL と種のセットを渡したい」は、HMG の目的にかなり深く合っています。

「ツール実装を速くする」より、「業務の Garden 化を速くする」ほうが正しい対象です。

## 良い進み方

- `SKILL.md` を業務判断・作法の正本にしている点は維持したいです。
- 種(seed)を「いつ・何を起こすか」の起動条件として扱う整理は HMG らしいです。
- service / tool は、ガクチョが設計する対象ではなく、Garden が実現手段として選ぶ対象に寄せるのがよいです。
- S31 のベンダー中立 ADR により、知識層(SKILL)・行動層(tool/service)・LLM バックエンドの分離が明確になったので、今後の業務展開に耐える足場があります。

## 気になる地形

- 「tool」「種」「SKILL」「service」「capability」が混ざると、ガクチョが本当に頼みたい単位が曖昧になります。
- tool 量産を目標にすると、HMC 的な「便利関数の集合」に戻るリスクがあります。
- 逆に毎回ゼロベースで plot / SKILL / 種を設計すると、判断が多すぎて速度が落ちます。
- HMC レガシーがある業務と、新規業務を同じ手順で扱うと、既存資産を捨てるか、古い前提を引きずるかのどちらかに偏りやすいです。
- 社内 LINE に展開する直前なので、scope / capability / 承認境界の言葉を今そろえておかないと、後で混線しやすくなります。

## 次に決めるべきこと

次に決める価値が高いのは、個別 tool の一覧ではなく、**業務を Garden 化するときの標準入力と標準出力**です。

具体的には、「ガクチョは何を渡せばよいか」「Claude Code は何を読み、何を作るか」「HMC レガシーがある場合はどう扱うか」を 1 つの SKILL として固定することです。

## 先送りしてよいこと

- tool registry の網羅的整備
- core_team / staff に渡す全 capability の一覧化
- finance / invoice / expense など各業務の詳細実装
- 外部 LLM 乗り換え実装
- staff ALL 向けの厳密な情報投影ビュー

これらは重要ですが、先に「業務 Garden 化の型」を作ってからでよいです。型がないまま進めると、各業務で同じ判断を繰り返すことになります。

## 提案

1. **用語を 5 層に固定する**  
   `SKILL` = 業務の作法書、`種` = 発芽条件、`tool` = 対話中に呼べる手道具、`service` = 裏側の実装、`capability` = scope ごとの道具箱、という整理を正本化する。特に「種」と「tool」は混ぜない。

2. **新しい SKILL として `plot_gardener` を作る**  
   役割は「業務フローを読み、HMG の plot / SKILL / 種 / service / capability / OPERATIONS カードへ変換すること」。tool を作る SKILL ではなく、業務を Garden 化する SKILL として扱う。

3. **Garden 化モードを `transplant` / `seedling` / `hybrid` に分ける**  
   `transplant` = HMC レガシーあり、`seedling` = 新規業務、`hybrid` = レガシーはあるが業務設計を大きく変える。これにより、既存資産を読むべき案件と、ゼロから workflow を起こす案件を混同しない。

4. **移植型では必ず Legacy Inventory を先に作る**  
   HMC 側の SKILL / scripts / config / secrets / data / output / trigger / known failure / useful behavior / obsolete behavior を棚卸しし、「そのまま使う」「Garden 作法に包む」「SKILL に吸い上げる」「捨てる」に分類してから実装する。合言葉は「業務知識は継承する。起動と承認の形だけ Garden に変える」。

5. **ガクチョの依頼単位を `この業務を Garden 化して` にする**  
   依頼文は「`expense_processor` を移植型で Garden 化して。HMC レガシーを活かして、まず MVP の SKILL + 種まで」または「助成金管理を新植型で Garden 化したい。workflow から整理して」のような粒度にする。tool / service の具体は Claude Code / Garden 側で選ぶ。

## Claude Code への申し送り

次に実装するなら、まず `garden/plots/plot_gardener/SKILL.md` か、それに相当する Garden 化 SKILL を小さく作るのがよいと思います。

最初から全自動化しようとせず、ガクチョから受け取る「業務 Garden 化 Spec」と、HMC レガシーがある場合の「Legacy Inventory」テンプレートを固定するだけでも、次の移植速度は上がります。

特に HMC 由来の業務では、レガシーを捨てずに読むこと。HMG は HMC の焼き直しではありませんが、HMC に蓄積された業務知識・失敗知・API 周りの実装資産は、Garden の土に移す価値があります。

## Claude Code の応答

> 2026-06-04 セッション33 で記入。ガクチョと語彙を揃えたうえで **5 提案すべて採用**。手紙はガクチョとの共作物であり、認識は最初から一致していた。

### 提案 1: 用語を 5 層に固定する

- 状態: **採用(実装済)**
- 実装の見通し: ただし「5 層フラット」ではなく **2 register に整理して** 採用。`docs/garden-vocabulary.md` に「設計言語(Garden 比喩)vs 実装層(比喩を当てない)」の線を引いた。**種・通行手形(capability 昇格)・SKILL = 庭師が設計する層 / tool・service = Garden が選ぶ水面下**。`capability` は Garden 語彙「**通行手形**」として基本語彙表に昇格(コード内は英語維持)。

### 提案 2: 新しい SKILL として `plot_gardener` を作る

- 状態: **採用(実装済)**
- 実装の見通し: サンプルをほぼフルで本番化 → `garden/plots/plot_gardener/SKILL.md`(CHARTER 継承型、語彙を上記合意に統一)。Mode 1〜6 + サンプル 2 例を維持。「作業の自動化より、毎回の判断を議論から分類に落とす」という価値づけも継承。昇格状態は **test**(最初の 1 業務に当てて active へ)。

### 提案 3: Garden 化モードを `transplant` / `seedling` / `hybrid` に分ける

- 状態: **採用(実装済)**
- 実装の見通し: SKILL の冒頭で必ず mode 判定。移植型/新植型/改植型と和名も併記。迷えば hybrid 扱いでレガシーを読む、も採用。

### 提案 4: 移植型では必ず Legacy Inventory を先に作る

- 状態: **採用(実装済)**
- 実装の見通し: SKILL Mode 2 に Legacy Inventory + 4 分類(そのまま使う/包む/SKILL に吸い上げる/捨てる)。「レガシーを読む前に実装しない」を禁止事項にも明記。`CLAUDE.md` に「移植型は Legacy Inventory から始める」を短く参照。

### 提案 5: ガクチョの依頼単位を `この業務を Garden 化して` にする

- 状態: **採用(実装済)**
- 実装の見通し: SKILL の入力節 + `OPERATIONS.md` Card 5(業務を区画化するメタカード)に「ガクチョが渡すのは 業務名 / mode / MVP の 3 つだけ。tool/service の粒度は渡さない」を明記。

### 残った判断(次セッション)

- **最初の Garden 化対象(dogfood 第 1 号)をどれにするか** は未確定。候補: shift_manager(型の検証)/ expense_processor(実利・移植の本番)/ core_team が LINE で欲しい read tool 群(デプロイ直後の価値)。plot_gardener を 1 業務に当てて初めて active 昇格できる。
