# HMG コンセプト

> このドキュメントは **育てる文書** — HMGの設計思想が進化するたびに更新する。

## 一行で言うと

**「人間中心(Cockpit)→AI中心(Garden)」** — 塚越さんが「庭師」となり、エージェント群が自律的に育つ生態系を運営する。

## HMCとHMGの根本的な違い

| | HMC (これまで) | HMG (これから) |
|---|---|---|
| 起点 | 塚越さんが指示 | エージェントが起動 |
| 業務の中央 | 塚越さんの操縦席 | 各エージェント(分散) |
| 計器類 | 塚越さんが読む | エージェントが読む。塚越さんは要約を見る |
| 不在時の挙動 | 停止 | 継続(必要なら通知) |
| 並列度 | 1業務ずつ | 多業務並列 |
| 塚越さんの席 | 操縦席(中央) | 庭の縁(俯瞰・剪定) |
| AIとの関係 | ユーザ ↔ ツール/Copilot | パートナー ↔ パートナー |

## 解決したい課題

1. **塚越さんが業務スタックのボトルネック** — タスク起動が全て人間側にあるため。
2. **並列処理がスムーズに進まない** — 起点が人間に集中するため、一つずつしか動けない。
3. **チームメンバーに開放できていない** — 当面は塚越さんワンユーザーで設計、将来チーム(LINE想定)に開放可能な構造を組み込む。
4. **コンテキストが散在している** — スタッフ・イベント・議事録・メールが各所にバラバラ。エージェントが自律判断するには統合された「土壌」が必要。

## エージェントが備えるべき4つの能力

1. **自律的起動** — 時間/イベント/状態変化を「種」として動き出せる
2. **コンテキスト自律取得** — 必要な「土壌」(マスター・履歴)に自分でアクセスできる
3. **判断・確認の規律** — 自律実行する判断と、塚越さんに「剪定」を仰ぐ判断を区別できる
4. **不足の明示** — 必要な情報が無い時、「何が足りないか」を提案できる

## 業務性質によって人間との関係が変わる

HMGは単一のエージェントモデルでなく、業務性質によって3つの関係性を使い分ける:

- **監督モデル**: AIが処理、塚越さん承認 (例: 請求書処理、経費登録)
- **パートナーモデル**: 塚越さんとAIが対等に議論しながら進める (例: クライアント管理、戦略議論)
- **庭師モデル**: AIが自律実行、塚越さんは結果を見るだけ (例: スタッフLINEリマインド、財務分析の定期実行)

## 育て方の規律

- 全部を一気に作らない。1区画ずつ育てる(HMCがfinance_importerから始めたのと同じ流儀)。
- 失敗・成功を**剪定**として塚越さんが反映、エージェントは学んでいく。
- 新概念を導入する時は必ず Garden語彙にマッピングする(語彙が無ければ追加する)。

## 影響を受けた発想

- Andrej Karpathy氏の **agentic engineering** — エージェントが自律的に動くソフトウェア観
- Anthropic/OpenAIが提唱する **人間中心 → AI中心の企業移行** — スモールビジネスで先行事例を作る
- 日本の **庭園・農のメタファ** — 育てる、剪定する、収穫する、季節を読む

## 制約と方針

- **ベンダー中立**: Claude/Gemini/GPT すべてからアクセスできる形式(markdown + sqlite + MCP server)を基本とする。プラットフォーム固有のメモリ機能に依存しない。
- **ローカル/VPS分担**: VPSはLINE等の窓口・24時間稼働のトリガー実行、ローカルはインタラクティブ作業・機密性の高い処理 — 詳細は別途設計。
- **既存HMCとの関係**: 当面実業務はHMC側で並行運用。HMGは設計・開発サンドボックスとして始め、段階的に実業務を移植する。
