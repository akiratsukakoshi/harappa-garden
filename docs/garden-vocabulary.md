# Garden 語彙 — HMG の設計言語

> このドキュメントは **育てる文書** — 新しい概念が必要になったら語彙を追加する。

「比喩を選ぶ意味は、語彙群を得ること」。HMGでは Garden 比喩を採用しているが、これは詩のためでなく **設計議論で「これは何?」と毎回考える代わりに、語彙から逆引きする** ためのもの。新しい概念が出てきたら必ずこの表に追加する。

## 基本語彙

| Garden語彙 | システム上の対応 | 実装の手触り |
|---|---|---|
| **庭師 (Gardener)** | 塚越さん | 戦略決定・最終承認・方向修正 |
| **庭 / Field** | HMG全体 | エージェント群の活動空間 |
| **区画 (Plot)** | 業務ドメイン | 財務区画・SNS区画・スケジュール区画 等 |
| **草木 / 作物 (Plant / Crop)** | 個別の業務タスク | 進行中の案件・処理 |
| **種 (Seed)** | トリガー | cron / イベント / 状態変化 |
| **苗床 (Nursery)** | 試行・育成中の業務 | プロトタイピング・新SKILL |
| **季節 (Season)** | 業務サイクル | 日次・週次・月次・年次 |
| **天気 (Weather)** | 外部状態 | カレンダー・メール・市況・天候 |
| **土壌 (Soil)** | コンテキスト基盤 | スタッフ/クライアント/イベントのマスター |
| **蔵 (Kura)** | 長期アーカイブ | 過去議事録・案件資料 |
| **番人 (Watcher)** | 監視エージェント | 異常検知・継続監視 |
| **菌糸 (Mycelium)** | 土壌維持エージェント | 分解・運搬・integrate・index 更新・関係性編み直し(裏方) |
| **収穫 (Harvest)** | 業務成果物 | レポート・提出物・記帳完了 |
| **剪定 (Pruning)** | 人間の介入 | 承認・却下・軌道修正 |
| **通行手形 (Capability)** | scope ごとの権限境界 | どの行動(tool)を誰が使えるか(master / core_team / staff) |
| **寄合 (Yoriai)** | エージェント間議論 | マルチエージェント協議 |

## 設計言語と実装層(2 つの register)

Garden 語彙は **比喩=庭師の思考言語**。「比喩を選ぶ意味は語彙群を得ること」の通り、庭師が「これは何?」を毎回考えずに済むための言語。だから比喩を当てられる概念だけが Garden 語彙に入る。

一方で、**比喩を当てない実装層の用語** がある。`tool` / `service` / `provider` は草でも種でも番人でもなく、ただの実装単位。これらは **Garden(Claude)が手段として選ぶ層=水面下** に置き、語彙表には載せない。庭師はこの層を設計しない。

| register | 語 | 誰の言語か |
|---|---|---|
| **設計言語(Garden 比喩)** | 区画・種・土壌・収穫・剪定・**通行手形** 等 | **庭師(ガクチョ)が考える層** |
| **実装層(比喩を当てない)** | tool・service・provider | **Garden(Claude)が手段を選ぶ層・水面下** |

業務を Garden 化するときの 5 層(SKILL / 種 / tool / service / 通行手形)は、この 2 register にこう割れる:

- **種** → 設計言語(発芽条件)
- **通行手形(capability)** → 設計言語(scope ごとの権限境界)
- **SKILL** → プロジェクト横断の確立語(HMC 継承。比喩ではないが定着済の作法書)
- **tool / service** → 実装層(水面下。庭師は粒度を決めない)

→ 庭師が設計するのは **区画・種・通行手形** の 3 つ。tool/service は Garden が選ぶ。
※ コード内では `capability` の英語表記を維持(`capabilities.py` 等)。設計議論では「通行手形」を使う。

## 役割の責務マップ(誰が何をするか)

| 役割 | 振る舞いの本質 |
|---|---|
| **庭師(Gardener)** | 意志を持つ。決める |
| **番人(Watcher)** | 監視する。告げる(目) |
| **菌糸(Mycelium)** | 分解する。運ぶ。整える(土の中の網目、見えない基盤) |
| **草木 / 木の精** | 表に出て話す(対話層 = ガクコ等) |

番人と菌糸の違い: 番人は「見て告げる」、菌糸は「分解して運ぶ」。責務が違うので独立した役割として立てる。

## 外部視点(庭の中の役者ではない)

庭の中の役者(植物・番人・菌糸・草木の精)とは別系統で、 **庭の縁の外** から地形を測る存在。

| 役割 | システム上の対応 | 実装の手触り |
|---|---|---|
| **測量士(Land Surveyor)** | codex(別系列 LLM)| 庭の外側から定期的に俯瞰し、目的との整合性・複雑化・正本関係を **手紙** で報告。実装はしない |

測量士は HMG の **外部観測者**。Claude Code が伴走しながら実装するのに対し、測量士はその外で「地形がいま全体としてどう見えるか」を返す。詳細: [`docs/surveyor/README.md`](surveyor/README.md)、[ADR: 測量士の導入](decisions/2026-06-02-surveyor-role.md)。

## 使い方の例

具体例(a) の Garden 語彙化:
> 「**財務区画**の**番人**が、メール(**天気**)を読み、**土壌**(スタッフマスター)を参照して候補を出し、塚越さん(**庭師**)に**剪定**を仰ぐ」

具体例(c) の Garden 語彙化:
> 「**スケジュール区画**の**番人**が **季節**(年間カレンダー) と **天気**(気象API) を継続監視し、必要な**剪定**を**庭師**経由でスタッフLINEに通知する」

具体例(d) の Garden 語彙化:
> 「**財務区画**で月初(**種**)に**収穫**(PL/CF レポート)を生成し、**庭師**に提示。SNS区画は**季節**と過去**収穫**を参照して投稿草案を生成」
