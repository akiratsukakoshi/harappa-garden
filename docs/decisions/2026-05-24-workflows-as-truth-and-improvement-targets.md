# workflow の扱い: 正本 かつ 改善対象

- **日付**: 2026-05-24
- **記録**: セッション5
- **決定者**: 塚越さん (庭師) / Claude (壁打ち相手)
- **ステータス**: 合意・適用開始(monthly-cycle.md を今セッションで書き直し。残り2本は次セッション以降)

## 背景

セッション5 で `shift_manager` SKILL の記述と workflows/monthly-cycle.md の実運用がズレていた問題が発覚した。原因の追跡から、2つの規律を同時に必要とすることが明らかになった:

1. **複数情報源の中で何を「正」とするか** — HMC SKILL マニュアル、Google Sheet の構造、運用ドキュメント、口頭知識など複数あるなかで、軸が必要
2. **「正本」を硬直化させない仕組み** — 正本にしてしまうと、現状の方法が固定化されて改善が止まる懸念がある

塚越さんの問題提起:
> workflowは正なのだが、改善可能性に溢れています。例えばコドモンからのデータ取得は本当に自動化できないのか(API,MCP は本当にない?)。自動化出来ないとしても、CSV からの入力は自動化可能(名前、時間の抽出なので可能)。

つまり **「目的は不変・方法は改善対象」** という二層構造を、workflow の記述自体に組み込む必要がある。

## 決定 1: workflow が正(skills や他データソースは追従)

- `garden/soil/workflows/` を業務実態の **single source of truth** とする
- HMC SKILL の記述・Google Sheet の構造・運用ドキュメント・口頭知識などとずれた場合、**workflow が勝つ**
- ずれを発見した場合は workflow を起点に他を直す(workflow を曲げない)
- ずれの発見と修正は、セッション内で Claude が気づいたタイミングで都度提案する

### 例(セッション5)

`shift_manager` SKILL の「新しい月になったら db_to_monthly.py でシートを作って」という記述と、workflows/monthly-cycle.md の「月次シートは3ヶ月に1回・企画会議前に生成」という実態がずれていた。後者(workflow)を正とし、SKILL の記述は古い記述として参考扱いに留めた。

## 決定 2: workflow は「目的が正・方法は改善対象」

workflow の各ステップは以下の二層で書く:

| 層 | 性質 | 変えてよいか |
|---|---|---|
| **目的 (Purpose)** | このステップが達成しなければならないこと | 不変(変えるならそれは別のステップ) |
| **現状の方法 (Current Method)** | 現在の手段。手作業・半自動・自動の混合可 | **変更可能**。簡略化・自動化の機会を常に探す |

加えて各ステップに **改善余地 (Improvement Hints)** セクションを設け、未検証の前提・自動化候補・過去の検討を記録する。

## 決定 3: workflow ステップの記述テンプレ(A 案)

```markdown
### N. ステップ名

- **時期**: いつ
- **担当**: 誰
- **目的**: このステップが達成しなければならないこと
  - 必須データ / 必須アウトプット
- **現状の方法**:
  1. ...
  2. ...
- **アウトプット**: 何が完成するか

#### 改善余地(Improvement Hints)

| # | 現状の方法 | 改善余地 | ステータス |
|---|---|---|---|
| ❓ | ... | ... | 未検証 |
| 💡 | ... | ... | 着手可能 |
| ✋ | ... | ... | 検討済(理由付き) |
| 🛠️ | ... | ... | 実装中 |
```

### ステータスマーカー

- **❓ 未検証** — 前提が確かめられていない(例: API/MCP の有無未確認)
- **💡 着手可能** — 改善方法が明確で、いま着手できる
- **✋ 検討済** — 一度検討した結果、現状維持の判断。**却下理由を併記**(同じ提案を繰り返さないため)
- **🛠️ 実装中** — 着手済・進行中

## 決定 4: 規律を明文化する場所

A 案の実効性を担保するため、原則を2箇所に書く:

1. `garden/soil/workflows/README.md` 冒頭 — workflow を開いた瞬間に必ず読む位置
2. `CLAUDE.md` — セッション開始時の初期コンテキストとして毎回読む位置

両方に「目的が正・方法は改善対象」と「改善余地セクションに気づいたら追記してから判断を仰ぐ」ことを書く。

## Claude への運用指示

- セッション開始時に workflow を読むたび、**改善余地セクションを必ず一読** する
- 新しい改善案を思いついたら、まず該当 workflow の改善余地表に追記してから塚越さんに提案する
- 「✋ 検討済」の改善案を再提案しない(却下理由を尊重)
- workflow と他情報源のずれを発見したら、workflow を正として他を直す提案をする
