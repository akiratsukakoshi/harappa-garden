# Instagram予約投稿 API制限問題の分析と回避策（セカンドオピニオン）

## 1. 課題の特定
- **エラー内容:** Meta Graph APIでの予約投稿（`published=false`）時に `(#3) User must be on whitelist` が発生。
- **根本原因:** 現在のMetaの仕様では、API経由の直接的なスケジュール機能は「Tech Provider（主要なSNS管理ツール事業者等）」としての厳格な審査・認証を通過したアプリに限定されている。
- **制約条件:** HMC（Harappa Management Cockpit）のような内部ツールにおいて、この認証を得ることはコスト・時間・要件の面で非現実的である。

## 2. 推奨される回避策：擬似スケジューリング・アーキテクチャ
Meta側の予約機能を使わず、自前のサーバー環境で実行タイミングを制御する「アプリケーション側でのスケジューリング」への移行を推奨する。

### システム構成案
1.  **データ生成（HMC / ローカル環境）:**
    - 投稿内容（画像URL、キャプション、予約時刻）を確定。
    - 24時間稼働するDockerサーバー側の待機キュー（DBまたはAPIエンドポイント）へデータを送信。
2.  **待機・監視（Dockerサーバー環境）:**
    - Pythonの `schedule` ライブラリや `Cron` を利用し、1分間隔で実行待機リストをスキャン。
    - 「現在時刻 ≧ 予約時刻」かつ「未投稿」のレコードを抽出。
3.  **投稿実行（Dockerサーバー環境）:**
    - 抽出したデータに基づき、既に動作確認済みである「即時投稿API」をコール。
    - コンテナIDの取得、および `media_publish` エンドポイントによる公開処理を完遂させる。

## 3. 本アプローチのメリット
- **審査のバイパス:** Metaの追加審査やTech Provider申請を一切必要とせず、現状の権限（`instagram_content_publish`）のみで実現可能。
- **資産の有効活用:** HMC で既に実装・検証済みの即時投稿ロジックをほぼそのまま流用できる。
- **拡張性:** 独自実装のため、Instagram公式APIが未対応の形式（ストーリーズの予約投稿の代替処理など）も、将来的に独自に組み込む余地が生まれる。

## 4. 実装上の留意点
- **トークン管理:** サーバー側でMetaのアクセストークンをセキュアに保持し、必要に応じて更新する仕組みが必要。
- **画像アクセシビリティ:** 即時投稿時、Metaのクローラーが指定の画像URLにアクセスできる状態（パブリックURL）を維持すること。
